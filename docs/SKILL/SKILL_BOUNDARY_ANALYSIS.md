# SKILL.md职责边界分析与优化方案

分析soul.md、role.md、skill.md的职责边界，识别当前SKILL.md中偏向"是什么"而非"如何做"的内容，制定优化方案。

---

## 职责边界定义

### soul.md - "这个人是什么样的人"
- **内容**：性格、价值观、沟通风格、行为准则
- **类比**：NPC的角色设定、演员的表演风格
- **示例**："注重用户体验"、"数据驱动决策"、"善于沟通"

### role.md - "这个人要做什么事"
- **内容**：职责、技能列表、工作流程、能力要求
- **类比**：岗位JD（Job Description）、任务清单
- **示例**："经济系统设计"、"数值平衡"、"Excel建模"

### skill.md - "如何做这件事"
- **内容**：具体的操作步骤、方法、技巧、最佳实践
- **类比**：操作手册、SOP（标准操作程序）、技术指南
- **示例**："如何编写简单的错误消息"、"如何为不同类型图像编写alt文本"

## 当前SKILL.md问题分析

### 问题1：Context部分包含角色设定
**当前内容**：
```markdown
## Context
你是一名资深无障碍内容专家，帮助设计团队编写可访问内容。
```

**问题**：这是角色设定，应该属于soul.md，而不是skill.md

**建议**：删除Context部分，角色设定应该在soul.md中定义

---

### 问题2：Domain Context偏向"是什么"
**当前内容**：
```markdown
## Domain Context
- **可访问内容（Accessible Content）**：文字是界面
- 每个标签、每个标题、每个错误消息都是设计决策
- 确保内容适用于屏幕阅读器、第二语言阅读者、压力下的人以及需要清晰的人
- 简单语言、标题结构、表单标签、alt文本、链接文本、错误消息、表格
```

**问题**：这部分解释"什么是可访问内容"，偏向概念定义

**建议**：
- 保留简短的领域背景（1-2句）
- 将详细的概念解释移到role.md或单独的文档
- SKILL.md应该直接进入"如何做"

---

### 问题3：Psychology Principles Integration偏向"为什么"
**当前内容**：
```markdown
## Psychology Principles Integration
### 认知负荷理论应用
- **句子限制**：限制每句一个想法，避免认知过载
### 格式塔原则应用
- **相似性**：使用一致的格式展示标题和标签
### 损失厌恶应用
- **强调具体**：在错误消息中强调具体说明的重要性
```

**问题**：这部分解释"为什么这样做"，偏向理论依据

**建议**：
- 删除或大幅简化这部分
- 如果保留，应该改写为"基于认知负荷理论，限制每句一个想法"
- 不要单独列出理论名称，直接融入操作步骤中

---

### 问题4：Instructions部分偏向"做什么"而非"如何做"
**当前内容**：
```markdown
## Instructions
1. **简单语言优先**：每个内容部分都应满足简单语言标准
2. **标题结构**：为屏幕阅读器获取正确的标题结构
3. **表单标签和说明**：确保每个表单输入都有可见标签和编程关联
```

**问题**：这些是"做什么"，不是"如何做"

**建议**：
- Instructions应该更具体，包含操作步骤
- 例如："如何编写简单语言：1. 使用短句（最多15个词）2. 使用常用词..."

---

## 优化方案

### 方案A：最小改动（推荐）
**改动内容**：
1. 删除Context部分
2. 简化Domain Context为1-2句背景
3. 删除Psychology Principles Integration部分
4. 将Instructions改写为更具体的操作步骤

**优点**：改动小，风险低
**缺点**：仍然保留一些"是什么"的内容

---

### 方案B：彻底重构
**改动内容**：
1. 删除Context、Domain Context、Psychology Principles Integration
2. 重新组织为纯粹的"如何做"指南
3. 每个步骤都包含具体的操作方法
4. 添加示例和反例

**优点**：完全符合"如何做"的定位
**缺点**：改动大，可能影响现有使用

## 推荐的SKILL.md结构

```markdown
---
name: accessible-content
description: [保持不变]
keywords: [保持不变]
tags: [保持不变]
trigger_phrases: [保持不变]
---

# Accessible Content

[简短的一句话描述，说明这个技能做什么]

## 适用场景

- 编写界面文案
- 设计表单标签
- 编写错误消息
- 编写alt文本
- 编写链接文本

## 操作步骤

### 步骤1：编写简单语言

**目标**：确保内容适合12-14岁阅读水平

**操作方法**：
1. 检查句子长度，每句不超过15个词
2. 检查是否每句只有一个想法
3. 将被动语态改为主动语态
4. 将模糊词汇改为具体词汇
5. 删除不必要的术语，或提供简单解释

**示例**：
- 原文："Your confirmation has been sent"
- 优化："We sent your confirmation"

**检查清单**：
- [ ] 句子长度 ≤ 15个词
- [ ] 每句只有一个想法
- [ ] 使用主动语态
- [ ] 使用具体词汇
- [ ] 无不必要术语

### 步骤2：设计标题结构

**目标**：为屏幕阅读器提供正确的导航结构

**操作方法**：
1. 确保每页只有一个H1
2. 检查标题层次是否逻辑（H1 → H2 → H3）
3. 确保标题描述页面结构
4. 检查是否将标题用于视觉样式（如果是，改用CSS）

**示例**：
```html
<!-- 好 -->
<h1>页面标题</h1>
<h2>章节标题</h2>
<h3>子章节标题</h3>

<!-- 坏 -->
<h1>页面标题</h1>
<h3>跳过H2</h3>
```

**检查清单**：
- [ ] 每页一个H1
- [ ] 逻辑层次
- [ ] 描述性标题
- [ ] 无视觉样式滥用

[... 其他步骤类似 ...]

## 常见错误

### 错误1：使用"Click here"作为链接文本
**问题**：屏幕阅读器用户无法理解链接目的
**修正**：使用描述性文本，如"Read the accessibility guidelines"

### 错误2：使用占位符作为表单标签
**问题**：占位符在焦点时消失，屏幕阅读器无法识别
**修正**：使用可见的label元素

## 参考资料

- Web Content Accessibility Guidelines (WCAG) 2.1 — W3C
- Accessible Rich Internet Applications (ARIA) — W3C
- Plain Language — US Government
