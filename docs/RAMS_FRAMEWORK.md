# RAMS / LMAIA – 角色-演员市场系统

RAMS: A Role-Actor Marketplace System for AI Agent Organization
Towards a Labor Market for AI Agents (LMAIA)

RAMS 是一个模型无关的通用智能体管理框架。它将 AI 能力抽象为 角色 (Role) 与 演员 (Actor) 两个正交维度，通过角色市场和演员市场实现岗位定义与劳动供给的解耦，并借助编排器 (Orchestrator) 实现任务驱动的多智能体协作与持续进化。

愿景：像管理人类团队一样管理 AI 智能体 —— 定义岗位、招聘演员、评估绩效、持续成长。

---

## 核心特性

- **角色与演员分离**：角色定义（Soul + Skills）与执行模型（GPT-5, Claude, Kimi...）解耦，同一角色可被不同模型扮演，同一模型可扮演不同角色。
- **双边市场机制**：
  - 角色市场：交易"岗位说明书"，支持购买、订阅、版本化、继承。
  - 演员市场：模型实例按需调用，实时价格、负载、信誉评分。
- **任务驱动编排**：结构化任务定义语言 (TDL)，基于 DAG 的依赖解析，支持并行、投票、路由等协作模式。
- **记忆与进化**：
  - 五层记忆架构（提示词记忆、会话存档、技能记忆、向量索引、用户画像）。
  - 后置学习闭环：自动从成功任务中提炼技能，根据用户反馈定向优化。
  - 容量驱动的主动压缩。
- **评估体系**：
  - 角色海选：标准化测试集评估（角色, 演员）组合。
  - 演员适配分：动态信誉系统。
  - 任务级校验（JSON Schema）。
- **跨领域适用**：游戏开发（首发应用）、金融、法律、医疗等。

## 核心概念详解

| 概念 | 说明 |
|------|------|
| 角色 (Role) | 标准化的 AI 岗位定义，包含人格设定 (Soul) 和技能堆栈 (Skills)，不绑定具体模型。 |
| 演员 (Actor) | 具体的大模型实例（如 GPT-5, Claude, 本地 Llama），可以扮演角色并执行任务。 |
| 角色市场 (Role Marketplace) | 交易角色定义的平台，支持购买、订阅、评价、版本升级。 |
| 演员市场 (Actor Marketplace) | 提供模型实例按需调用，记录实时价格、负载、信誉评分。 |
| 编排器 (Orchestrator) | 中央调度器，解析任务 DAG，匹配角色与演员，管理记忆与进化。 |
| 任务定义语言 (TDL) | 结构化 JSON/YAML 格式，包含输入输出 Schema、依赖关系、能力要求。 |
| 角色海选 (Role Audition) | 使用标准测试集评估（角色, 演员）组合的表现，生成适配度矩阵。 |
| 后置学习 (Post-Task Learning) | 任务完成后自动提炼技能或优化角色定义，实现持续进化。 |

## 框架的核心组成

| 层次 | 组件 | 功能说明 |
|------|------|----------|
| 角色定义层 | Soul + Skills + 变体 | 标准化描述 AI 的职责、行为准则、能力边界，支持版本化与继承 |
| 角色市场 | 角色注册、搜索、交易、评价 | 类似招聘网站的职位库，用户可以发布、购买、订阅角色模板 |

| 编排器 | 任务解析、角色匹配、演员调度、记忆管理 | 解析任务 DAG，自动为每个子任务选择合适的角色与演员，并行执行，管理长期记忆 |

## 如何"从角色角度组织 AI"

传统方式以模型为中心：用户先选模型（如 GPT-5），然后为它写提示词。我们的框架以角色为中心：

1. **先定义角色**：例如"游戏数值策划"角色，包含其人格设定、所需技能（数值平衡、Excel 建模）、能力标签（高推理、工具使用）。
2. **角色市场提供模板**：用户可以直接购买或继承现有角色，无需从零设计。
3. **角色绑定演员**：编排器根据任务需求，从演员市场动态选择最适合的模型来扮演该角色。同一个角色可由不同模型扮演，同一个模型可扮演不同角色。
4. **角色拥有独立记忆和技能库**：所有扮演该角色的演员共享角色的长期记忆和进化出的技能，实现能力持续积累。

结果：用户只需要说"我需要一个 Unity 工程师角色"，而不是"我要用 Claude 写一段 Unity 代码"。AI 的组织方式从"工具调用"转变为"岗位配置"。

## 如何"评估 AI 在各类任务中的能力"

我们的评估体系是多层次、多维度的：

| 评估层次 | 评估对象 | 方法 | 产出 |
|----------|----------|------|------|
| 角色海选 | 角色定义 + 候选演员 | 标准测试集（如"实现角色移动"、"设计经济数值"）自动执行，打分 | 角色-演员适配度矩阵、角色综合评分 |
| 任务级评估 | 单个任务执行结果 | 输出 Schema 校验、单元测试、用户反馈 | 任务成功/失败、质量评分（0-10） |
| 演员信誉 | 演员在特定角色上的历史表现 | 加权平均（质量、成本、延迟、成功率） | 演员适配分，动态更新 |
| 角色市场评价 | 角色模板本身 | 用户购买后的评分与评论 | 星级、好评率、使用量 |

关键创新：评估不是针对固定模型，而是针对（角色, 演员）组合，且评估结果会反馈给编排器，用于未来的智能调度。

---

## 闭环：从评估到进化

```
任务执行 → 评估（输出校验+用户反馈）→ 更新演员适配分
                                      ↓
                              若重复模式或用户修正
                                      ↓
                              后置学习 → 创建/修改角色技能
                                      ↓
                              角色进化 → 下次所有演员受益
```

这个闭环使得框架不仅能评估当前能力，还能持续提升角色能力，形成"越用越强"的正向循环。

## 角色进化与演员适配

### 核心结论：角色在进化，演员在适配

- **进化（能力增长）** → 发生在角色定义上（Soul + Skills 静态文件变化）
- **适配（匹配度变化）** → 发生在演员上（信誉分、成本、延迟等运行时指标变化）

---

### 一、角色进化：静态定义的变化

**进化的本质**：角色的 Soul（人设） 和 Skills（技能堆栈） 内容被修改或扩充。

**谁触发**：后置学习闭环（Post-Task Learner）分析任务执行过程后，更新角色定义文件。

**具体例子**：

| 时间 | 角色 client_unity 的 Skills | 说明 |
|------|---------------------------|------|
| 初始 | [csharp, unity_development, ui_toolkit] | 只有基础技能 |
| 经过10次"实现2D角色移动"任务后 | [csharp, unity_development, ui_toolkit, unity_2d_movement_template] | 新技能被创建并添加到角色定义中 |
| 用户反馈"生成的代码缺少异常处理"后 | patch_skill 修改 unity_2d_movement_template 的内容 | 技能内容被优化 |

**关键点**：
- 这个进化后的角色定义，所有后续扮演该角色的演员（无论Claude还是GPT-5）都能获得新技能。
- 因为技能是角色定义的一部分，演员只是加载并执行它。

---

### 二、演员适配：运行时指标的变化

**适配的本质**：演员的信誉分、成本、延迟、成功率等指标更新，影响编排器是否选择它来扮演某个角色。

**谁触发**：每次任务执行后，编排器记录该演员在该角色上的表现（质量、耗时、成本），更新演员档案中的 suitability_for_role 矩阵。

**具体例子**：

| 演员 | 在角色 client_unity 上的适配度 | 变化原因 |
|------|-----------------------------|----------|
| Claude | 9.5 → 9.6 | 连续10次任务成功，质量评分上升 |
| GPT-5 | 8.8 → 8.5 | 最近两次任务出现代码错误，扣分 |
| Kimi | 7.5 → 7.8 | 价格下降20%，成本效率权重提高 |

**关键点**：
- 演员不会获得新技能（模型本身的参数不变，除非重新训练）。
- 演员的适配度变化不会修改角色定义，只影响调度时的选择概率。
- 一个演员可以适配多个角色（例如Claude既适合 client_unity，也适合 backend_dev），分别有不同的适配分。

---

### 三、记忆压缩：角色级还是演员级？

**答**：角色级（更准确地说，是角色实例级，但共享同一个角色定义的不同实例可以有独立的记忆）。

- **MEMORY.md / USER.md**：存储在角色定义目录下（例如 ~/.agentverse/roles/client_unity/memory.md）。所有演员扮演该角色时，读取同一份记忆。
- **会话存档**：按（角色, 用户, 项目）隔离，但通常绑定到角色实例，而不是演员。
- **向量索引**：角色级的长期知识库。

**为什么这样设计？**
- 记忆应该是角色的长期经验，与谁扮演无关。Claude演这个角色时积累的"用户偏好蓝色主题"，GPT-5演时也应该知道。
- 如果记忆绑定到演员，换模型就要重新学习，效率低。

---

### 四、对比表格

| 维度 | 角色进化 | 演员适配 |
|------|----------|----------|
| 变化对象 | 角色定义（Soul + Skills 文件） | 演员档案（信誉、价格、延迟） |
| 变化内容 | 新增/修改技能、优化人设描述 | 适配分、成功率、成本、负载 |
| 触发机制 | 后置学习闭环（任务成功/用户反馈） | 每次任务执行后的指标记录 |
| 持久性 | 永久，直到下次被修改 | 动态，每次调用后都会微调 |
| 影响范围 | 所有扮演该角色的演员 | 仅影响该演员被选择的概率 |
| 是否可交易 | 是（进化后的角色可上传市场） | 否（演员市场的信誉是平台内部数据） |
| 例子 | client_unity 获得新技能 unity_2d_movement_template | Claude 在 client_unity 角色上的适配分从9.2升到9.4 |

---

### 五、两者如何协作？

```
用户反馈："这次生成的移动代码缺少跳跃功能"
    ↓
编排器记录负反馈 → 降低Claude在该任务类型上的适配分（演员适配）
    ↓
同时触发后置学习组件 → 分析对话，发现"跳跃"常被遗漏
    ↓
调用 patch_skill → 修改角色 client_unity 的技能 unity_2d_movement_template
    ↓
进化后的角色定义被保存
    ↓
下次任何演员扮演 client_unity 时，都会加载改进后的技能，不再遗漏跳跃
```

**结果**：
- 角色进化了（技能更完善）
- 演员Claude的适配分暂时下降（因为这次表现不好），但未来如果其他演员用进化后的技能表现更好，Claude的适配分可能回升（如果它能正确使用新技能）。

---

### 总结一句话

**角色是剧本，进化的是剧本本身（人设+技能）；演员是临时工，适配的是雇佣记录（信誉+成本）。**

- **角色进化**：改剧本 → 所有演员受益
- **演员适配**：改面试评分 → 影响谁被录用

## 框架的完整性与独特性

| 传统 AI 应用 | 本框架 |
|--------------|--------|
| 以模型为中心，每次重新设计提示词 | 以角色为中心，角色定义可复用、可交易 |
| 评估模型通用能力（MMLU、HumanEval） | 评估（角色, 演员）在特定任务上的匹配度 |
| 无标准化岗位描述 | 角色市场提供类似招聘网站的标准化职位 |
| 记忆和技能绑定到单次会话 | 角色拥有长期记忆和可进化的技能堆栈 |
| 模型选择靠人工试错 | 编排器基于适配度矩阵动态调度 |

## 以终为始：RAMS/LMAIA 框架的终局分析与倒推路径

"以终为始" 意味着从最终愿景反向推导出当前需要采取的行动。我们设想的终点是一个全球性的 AI 劳动力市场——任何人都可以发布 AI 岗位（角色），任何模型都可以作为劳动力（演员）被调用，任何团队都可以像组建人类团队一样组建 AI 团队。

### 终局愿景（End State）

- **角色市场**：拥有超过 100 万个角色定义，覆盖游戏、金融、医疗、法律、教育等主流行业，角色定义语言（RDL）成为 ISO 标准。
- **演员市场**：接入全球主流大模型（OpenAI、Anthropic、Google、Meta、开源社区），以及无数垂直领域微调模型，实时竞价、动态匹配。
- **编排器**：成为企业级 AI 操作系统的核心组件，与 LangChain、AutoGen 等框架深度集成，支持本地、云端、边缘部署。
- **生态**：有大量第三方开发者靠创作角色、开发技能、提供演员服务获得收入，形成正向经济循环。
- **用户**：任何规模的企业（从独立开发者到财富 500 强）都能在几分钟内搭建专属 AI 团队，成本仅为人类团队的 1/10。

### 关键决策点（以终为始的优先级）

| 决策 | 终局影响 | 当前建议 |
|------|----------|----------|
| 是否开源？ | 开源有助于标准化和生态，但可能降低商业收入 | 核心编排器开源，市场平台闭源（SaaS） |
| 先做角色市场还是演员市场？ | 两者互相依赖，但角色市场更容易启动 | 先做角色市场（定义可静态提供），演员市场初期用用户自备 API key 代替 |
| 是否支持本地模型？ | 本地模型对数据敏感企业很重要 | 在 MVP 阶段不支持，但架构预留接口 |
| 角色海选自动化？ | 自动化评估是信任基石 | P0 阶段手动评估，P1 实现自动化脚本 |

```

     |              |              |             |            |           |           |
  概念验证      单角色单模型    5角色2模型   20角色3模型  100角色    1000角色   100万角色
```
