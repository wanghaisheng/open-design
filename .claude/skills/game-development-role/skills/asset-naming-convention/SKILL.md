# Asset Naming Convention

设计游戏资产命名规范，包括纹理、模型、材质、动画、音频等类型的命名规则、文件夹结构和元数据管理，确保团队协作效率和资产管理的一致性。

## 适用场景

- 新项目启动时建立统一的资产命名规范
- 现有项目资产命名混乱，需要重构规范
- 团队协作中出现资产查找困难、命名冲突问题
- 需要建立工业化美术管线，提升资产管理效率

## 操作步骤

### 步骤1：定义资产类型分类和前缀

**目标**：建立清晰的资产类型分类体系，为每种类型分配唯一前缀

**操作方法**：
1. 列出项目中所有资产类型（纹理、模型、材质、动画、音频、特效、UI、灯光等）
2. 为每种类型分配简短、唯一的前缀（如 T_ 纹理、SM_ 静态模型、SK_ 骨骼模型）
3. 定义常用后缀（如 _D 漫反射、_N 法线、_RMA 粗糙度-金属-AO）
4. 建立资产类型对照表

**检查清单**：
- [ ] 资产类型分类完整
- [ ] 前缀简短且唯一（2-3字符）
- [ ] 后缀符合行业标准（PBR、LOD等）
- [ ] 对照表清晰易懂

### 步骤2：设计命名格式规则

**目标**：建立统一的命名格式，确保机器可读和一致性

**操作方法**：
1. 定义通用格式：[类型前缀]_[主体描述]_[渠道/LOD]_[变体]
2. 制定命名规则：
   - 全英文，避免 Unicode/特殊字符
   - 长度限制（< 50字符）
   - 使用 snake_case 或 PascalCase
   - 避免空格
3. 定义版本控制规则（_01、_A、v001）
4. 提供命名示例

**检查清单**：
- [ ] 命名格式清晰
- [ ] 规则简单易记
- [ ] 示例丰富且实用
- [ ] 版本控制规则明确

### 步骤3：设计文件夹结构

**目标**：建立便于协作和打包的文件夹组织方式

**操作方法**：
1. 选择组织方式：按功能分层（推荐大项目）或按类型子分
2. 设计目录树结构：
   - Art/（角色、武器、环境、UI、FX）
   - Audio/（音效、音乐、语音）
   - _Publish/（最终导出）
3. 定义子文件夹命名规范（PascalCase）
4. 设计工作区隔离（_wip 工作文件、_publish 发布文件）

**检查清单**：
- [ ] 目录结构逻辑清晰
- [ ] 命名规范统一
- [ ] 工作区隔离明确
- [ ] 版本控制友好

### 步骤4：设计元数据标签系统

**目标**：为资产附加结构化元数据，支持智能搜索和自动化

**操作方法**：
1. 定义标准标签键值对：
   - LOD（High/Low/Medium）
   - Platform（PC/Mobile）
   - Usage（Character/UI/ENV）
   - Author（责任人）
   - Version（版本追踪）
   - Dependencies（依赖关系）
2. 设计元数据格式（JSON/XML）
3. 规划元数据附加方式（导出钩子、手动添加）
4. 设计搜索和过滤机制

**检查清单**：
- [ ] 标签键值对完整
- [ ] 元数据格式标准化
- [ ] 附加方式可行
- [ ] 搜索机制高效

### 步骤5：制定实施和验证流程

**目标**：确保命名规范在团队中有效执行

**操作方法**：
1. 编写规范文档和培训材料
2. 开发或配置自动化工具：
   - Editor 插件警告
   - 批量重命名脚本
   - CI/CD 校验
3. 建立审核流程（Code Review 资产命名）
4. 定期审计和优化

**检查清单**：
- [ ] 规范文档完整
- [ ] 自动化工具就绪
- [ ] 审核流程建立
- [ ] 持续优化机制

## 操作注意事项

### 一致性优先

1. 命名规范必须在项目启动时确定并文档化
2. 避免中途更改规范，如需更改需团队共识
3. 同一资产在不同引擎中使用相同命名

### 普通人视角

1. 提供普通表达对照表（专业术语 vs 普通表达）
   - Texture → 皮肤/表面图案
   - Model → 人物/物体形状
   - Animation → 动作序列
   - Prefab → 现成组合件
2. 使用电影一帧类比帮助非专业人士理解

### 工业化思维

1. 从小项目起步，逐步扩展到中大型项目
2. 自动化优先：脚本强制校验，提升效率 50%+
3. 结合 DAM 系统（如 Artstash）避免孤岛

## 常见错误

### 错误1：命名不一致

**问题**：团队成员使用不同命名风格，导致查找困难

**修正**：在步骤1制定统一规范，步骤5配置自动化工具强制校验

### 错误2：文件夹结构混乱

**问题**：资产散落在不同位置，难以打包和版本控制

**修正**：在步骤3设计清晰的目录树，使用 _wip/_publish 隔离

### 错误3：缺少元数据

**问题**：资产缺少关键信息，无法智能搜索和自动化处理

**修正**：在步骤4设计元数据标签系统，通过导出钩子强制附加

### 错误4：规范过于复杂

**问题**：规范太复杂，团队难以执行

**修正**：从小项目起步，逐步扩展；优先核心类型，简化规则
