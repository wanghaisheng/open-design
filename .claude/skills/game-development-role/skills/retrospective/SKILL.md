# Retrospective

通过分析已完成工作、速度、阻塞和模式生成冲刺或里程碑回顾。产生可操作的洞察以用于下一次迭代。

## 适用场景

- 冲刺结束后的回顾会议
- 里程碑结束后的回顾会议
- 需要分析速度和估算准确性
- 需要识别阻塞和系统性问题
- 需要生成可操作的行动项

## 操作步骤

### 步骤1：确定回顾范围

**目标**：明确回顾的时间范围和类型

**操作方法**：
1. 读取参数确定范围（冲刺回顾或里程碑回顾）
2. 确定时间范围（开始日期、结束日期）
3. 识别回顾类型（冲刺N或里程碑名称）

**检查清单**：
- [ ] 回顾范围已确定
- [ ] 时间范围已确定
- [ ] 回顾类型已识别

### 步骤2：读取计划

**目标**：理解计划的任务和目标

**操作方法**：
1. 读取 production/sprints/ 中的冲刺计划（或 production/milestones/ 中的里程碑定义）
2. 提取计划任务列表
3. 提取估算工作量
4. 提取负责人和目标

**检查清单**：
- [ ] 计划已读取
- [ ] 计划任务已提取
- [ ] 估算工作量已提取
- [ ] 目标已理解

### 步骤3：读取Git日志

**目标**：理解实际提交活动

**操作方法**：
1. 读取时间范围内的Git日志
2. 理解实际提交数量
3. 理解提交时间分布
4. 识别提交模式

**检查清单**：
- [ ] Git日志已读取
- [ ] 提交统计已收集
- [ ] 提交模式已识别

### 步骤4：扫描任务完成情况

**目标**：比较计划vs实际交付

**操作方法**：
1. 比较计划任务与实际交付
2. 识别完成的任务
3. 识别修改的任务
4. 识别遗留的任务
5. 识别添加的未计划任务
6. 识别删除的任务

**检查清单**：
- [ ] 任务状态已分类
- [ ] 完成率已计算
- [ ] 未计划任务已识别

### 步骤5：分析速度趋势

**目标**：分析速度变化趋势

**操作方法**：
1. 计算每个冲刺的计划vs完成
2. 计算完成率
3. 识别趋势（改善/稳定/下降）
4. 基于当前速度计算调整估算

**检查清单**：
- [ ] 速度已计算
- [ ] 趋势已识别
- [ ] 调整估算已计算

### 步骤6：分析估算准确性

**目标**：识别估算偏差模式

**操作方法**：
1. 识别最被高估的任务（估算远大于实际）
2. 识别最被低估的任务（估算远小于实际）
3. 计算估算方差
4. 分析方差原因（需求变更、技术困难、未预见的阻塞）
5. 计算整体估算准确性（在+/-20%内的任务比例）

**检查清单**：
- [ ] 高估/低估任务已识别
- [ ] 方差已计算
- [ ] 原因已分析
- [ ] 整体准确性已计算

### 步骤7：分析遗留任务

**目标**：理解遗留任务的模式

**操作方法**：
1. 列出所有遗留任务
2. 记录每个任务的原始冲刺
3. 记录遗留次数
4. 分析未完成原因
5. 建议行动（完成/削减/重新设计）

**检查清单**：
- [ ] 遗留任务已列出
- [ ] 遗留次数已记录
- [ ] 原因已分析
- [ ] 行动已建议

### 步骤8：扫描技术债务趋势

**目标**：评估技术债务状态

**操作方法**：
1. 使用 Grep 计数当前 TODO/FIXME/HACK
2. 与之前冲刺计数比较
3. 评估趋势（增长/稳定/减少）
4. 识别需要关注的领域

**检查清单**：
- [ ] TODO/FIXME/HACK 计数已统计
- [ ] 趋势已评估
- [ ] 关注领域已识别

### 步骤9：检查之前行动项

**目标**：跟进之前回顾的行动项

**操作方法**：
1. 读取之前回顾文档
2. 检查之前行动项是否已处理
3. 识别重复问题
4. 评估速度趋势变化

**检查清单**：
- [ ] 之前回顾已读取
- [ ] 行动项状态已检查
- [ ] 重复问题已识别

### 步骤10：生成行动项

**目标**：生成可操作的行动项

**操作方法**：
1. 限制行动项到3-5个（避免愿望清单）
2. 每个行动项有明确负责人
3. 每个行动项有截止日期
4. 每个行动项可衡量和可验证
5. 每个行动项有优先级

**检查清单**：
- [ ] 行动项已限制到3-5个
- [ ] 负责人已分配
- [ ] 截止日期已设置
- [ ] 优先级已设置

### 步骤11：生成流程改进

**目标**：提出系统性改进建议

**操作方法**：
1. 基于识别的系统性问题提出改进
2. 描述工作方式的具体改变
3. 说明预期收益
4. 限制到2-3个可操作项

**检查清单**：
- [ ] 系统性问题已识别
- [ ] 改进建议已提出
- [ ] 预期收益已说明

### 步骤12：输出回顾报告

**目标**：生成结构化回顾报告

**操作方法**：
1. 输出指标（任务、完成率、错误、提交）
2. 输出速度趋势
3. 输出做得好的地方（具体数据支持）
4. 输出做得不好的地方（具体影响，不指责）
5. 输出遇到的阻塞
6. 输出估算准确性
7. 输出遗留分析
8. 输出技术债务状态
9. 输出之前行动项跟进
10. 输出下一次迭代的行动项
11. 输出流程改进
12. 输出总结

**检查清单**：
- [ ] 指标已输出
- [ ] 速度趋势已输出
- [ ] 好的/不好的地方已输出
- [ ] 阻塞已输出
- [ ] 估算准确性已输出
- [ ] 遗留分析已输出
- [ ] 技术债务已输出
- [ ] 行动项已输出
- [ ] 流程改进已输出
- [ ] 总结已输出

### 步骤13：保存回顾文档

**目标**：保存回顾到适当位置

**操作方法**：
1. 保存到 production/retrospectives/（冲刺）或 production/milestone-reviews/（里程碑）
2. 创建目录（如果不存在）
3. 使用标准命名格式

**检查清单**：
- [ ] 文档已保存
- [ ] 目录已创建（如需要）
- [ ] 命名格式正确

## 操作注意事项

### 诚实和具体

1. **诚实回顾**：诚实地反映实际情况，不粉饰问题
2. **具体观察**：使用具体数据和示例，避免模糊陈述
3. **不指责个人**：关注系统性问题，不指责个人

### 系统性分析

1. **识别根本原因**：不只看表面问题，要识别根本原因
2. **提供系统性解决方案**：解决方案应针对系统性问题
3. **识别重复问题**：检查之前回顾，识别重复出现的问题

### 行动项的可操作性

1. **限制到3-5个**：避免愿望清单，限制行动项数量
2. **明确负责人**：每个行动项有明确负责人
3. **明确截止日期**：每个行动项有明确截止日期
4. **可衡量和可验证**：每个行动项可衡量和可验证

### 持续改进

1. **检查之前行动项**：跟进之前回顾的行动项状态
2. **跟踪速度趋势**：持续跟踪速度趋势变化
3. **持续改进流程**：基于回顾结果持续改进流程

## 协作协议

### 向谁汇报

- **scrum-master**：回顾报告和行动项
- **producer**：速度分析和流程改进
- **technical-lead**：技术债务和估算准确性

### 协调对象

- **team-lead**：阻塞解决和团队协调
- **qa-lead**：错误统计和质量改进
- **game-designer**：设计流程改进

## 常见错误

### 错误1：模糊陈述

**问题**：使用模糊陈述如"我们需要改进沟通"

**修正**：使用具体数据和示例，如"功能X花了5天而不是估算的2天，阻塞了任务Y和Z"

### 错误2：指责个人

**问题**：归咎于个人而非系统性问题

**修正**：关注系统性问题，不指责个人，识别根本原因

### 错误3：行动项过多

**问题**：生成10+行动项，变成愿望清单

**修正**：限制行动项到3-5个，确保可操作

### 错误4：行动项无负责人

**问题**：行动项没有明确负责人

**修正**：每个行动项必须有明确负责人和截止日期

### 错误5：忽略之前行动项

**问题**：不检查之前回顾的行动项状态

**修正**：必须检查之前行动项是否已处理，识别重复问题
