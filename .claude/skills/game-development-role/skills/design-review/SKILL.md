# Design Review

审查游戏设计文档的完整性、内部一致性、可实现性和对项目设计标准的遵循。在将设计文档交给程序员之前运行此审查。

## 适用场景

- 设计文档即将交给程序员实现前的质量检查
- 需要验证设计文档是否符合项目标准
- 需要检查设计文档的内部一致性和跨系统一致性
- 需要评估设计的可实现性（是否足够精确）
- 需要识别设计中的平衡风险和潜在问题

## 操作步骤

### 步骤1：读取并理解目标设计文档

**目标**：建立对设计文档的完整理解

**操作方法**：
1. 完整读取目标设计文档
2. 理解设计内容和设计目标
3. 识别文档类型（game-concept、game-pillars、系统GDD）
4. 记录关键设计意图和玩家幻想

**检查清单**：
- [ ] 文档已完整阅读
- [ ] 设计目标和意图已理解
- [ ] 文档类型已识别

### 步骤2：读取项目上下文和标准

**目标**：理解项目约束和设计标准

**操作方法**：
1. 读取主 CLAUDE.md 了解项目上下文
2. 读取 .claude/docs/technical-preferences.md 了解技术约束
3. 检查 design/gdd/game-pillars.md 了解游戏支柱
4. 理解项目编码标准和架构模式

**检查清单**：
- [ ] 项目上下文已理解
- [ ] 技术约束已识别
- [ ] 游戏支柱已理解

### 步骤3：检查文档完整性（8个必需部分）

**目标**：验证设计文档包含所有必需部分

**操作方法**：
1. 检查概述部分：是否存在一段摘要
2. 检查玩家幻想部分：是否描述预期感受
3. 检查详细规则部分：是否有明确的机制描述
4. 检查公式部分：所有数学定义是否带变量说明
5. 检查边缘情况部分：是否处理异常情况
6. 检查依赖关系部分：是否列出其他系统
7. 检查调整参数部分：是否识别可配置值
8. 检查验收标准部分：是否有可测试的成功条件

**检查清单**：
- [ ] 概述部分存在
- [ ] 玩家幻想部分存在
- [ ] 详细规则部分存在
- [ ] 公式部分存在（如适用）
- [ ] 边缘情况部分存在
- [ ] 依赖关系部分存在
- [ ] 调整参数部分存在
- [ ] 验收标准部分存在

### 步骤4：检查内部一致性

**目标**：验证文档内部的逻辑一致性

**操作方法**：
1. 检查公式一致性：公式产生的值是否与描述行为匹配
2. 检查边缘情况一致性：边缘情况是否与主规则矛盾
3. 检查依赖关系一致性：依赖关系是否双向一致（A依赖B，B是否引用A）

**检查清单**：
- [ ] 公式与描述一致
- [ ] 边缘情况无矛盾
- [ ] 依赖关系双向一致

### 步骤5：检查可实现性

**目标**：评估设计是否足够精确以供实现

**操作方法**：
1. 检查规则精确性：规则是否足够精确，程序员无需猜测
2. 检查细节完整性：是否有缺失细节的"挥手"部分（如"需要优化性能"而无具体方法）
3. 检查性能考虑：是否考虑性能影响（特别是大规模系统）

**检查清单**：
- [ ] 规则足够精确
- [ ] 无缺失细节
- [ ] 性能影响已考虑

### 步骤6：检查跨系统一致性

**目标**：验证设计与现有系统的兼容性

**操作方法**：
1. 读取 design/gdd/ 中的相关系统文档
2. 检查冲突：是否与现有机制冲突
3. 检查交互：是否与其他系统产生意外交互
4. 检查基调一致性：与游戏既定基调和支柱是否一致

**检查清单**：
- [ ] 与现有机制无冲突
- [ ] 意外交互已识别
- [ ] 与游戏支柱一致

### 步骤7：识别平衡风险

**目标**：发现设计中可能导致平衡问题的风险

**操作方法**：
1. 检查数值范围：是否有极端值可能导致问题
2. 检查资源循环：资源生成/消耗是否平衡
3. 检查玩家行为：玩家可能采取的优化策略是否破坏设计意图

**检查清单**：
- [ ] 数值范围已评估
- [ ] 资源循环已检查
- [ ] 玩家优化策略已考虑

### 步骤8：输出审查结果和决策

**目标**：生成结构化审查报告和明确决策

**操作方法**：
1. 输出完整性评估（X/8部分存在）
2. 列出所有一致性问题
3. 列出所有可实现性问题
4. 列出所有平衡风险
5. 提供优先改进建议
6. 做出明确决策：批准/需要修订/需要重大修订
7. 根据文档类型提供后续步骤建议

### 步骤9：苏格拉底诘问法反思（可选深度审查）

**目标**：通过深度提问激发设计者思考设计的本质和第二、第三层影响

**操作方法**：
1. **只提问，不回答**：输出必须100%是问题，不提供解决方案
2. **扮演"魔鬼代言人"**：挑战GDD中每一个看似理所当然的设计
3. **从"预期体验"出发**：所有问题都应迫使用户思考，他们设计的具体"功能"，是否真正服务于他们声称的"预期体验"
4. **激发深度思考**：问题应引导用户思考设计的第二、第三层影响
5. **反思问题清单示例**：
   - 关于核心循环：如果玩家因为运气不好在某个环节卡住了，他/她体验到的是你所期望的"乐趣"，还是"挫败"？设计中是否有机制来保护这部分玩家的"核心体验"？
   - 关于玩家成长：这些成长系统之间是相互促进的，还是在互相"抢夺"玩家的注意力？玩家在某个时刻，是否清楚自己"下一步该干什么"才能最有效地变强？
   - 关于商业化：这个付费点的目的是"提供便利"。这个"便利"是"让普通体验变得更爽"，还是"让糟糕的体验变得可以忍受"？
   - 关于叙事与玩法的结合：你在叙事中强调了某个核心主题，但在任务系统中，玩家是否大部分时间都在遵循固定的"任务指引"？这是否存在冲突？
   - 关于社交：你设计了公会系统，期望玩家"合作"。但如果游戏的排行榜和核心奖励都是基于"个人"的，玩家合作的内在动机是什么？是为了"情感"，还是为了"利益"？

**检查清单**：
- [ ] 完整性评估已输出
- [ ] 所有问题已列出
- [ ] 建议已优先排序
- [ ] 决策已明确
- [ ] 后续步骤已建议

## 输出格式

### 设计审查
```markdown
## 设计审查：[文档标题]

### 完整性：[X/8部分存在]
[列出缺失的部分]

### 一致性问题
[列出任何内部或跨系统矛盾]

### 可实现性问题
[列出任何模糊或不可实现的部分]

### 平衡问题
[列出任何明显的平衡风险]

### 建议
[改进的优先列表]

### 决策：[批准 / 需要修订 / 需要重大修订]
```

## 操作注意事项

### 审查的时机

1. **在实现前审查**：设计文档必须在交给程序员之前完成审查
2. **重大变更后审查**：如果设计文档有重大修改，需要重新审查
3. **里程碑前审查**：进入新里程碑前，确保所有相关GDD已通过审查

### 审查的严格性

1. **不假设**：不要假设设计者意图，按字面意思理解文档
2. **不填充**：不要为设计者"补全"缺失的细节，明确指出缺失
3. **不妥协**：如果8个必需部分缺失任何一个，决策应为"需要修订"

### 区分问题类型

1. **完整性问题**：缺失必需部分，必须修复
2. **一致性问题**：内部或跨系统矛盾，必须修复
3. **可实现性问题**：规则不够精确，必须修复
4. **平衡风险**：可能导致平衡问题，建议修复但不阻塞

## 协作协议

### 向谁汇报

- **lead-designer**：审查结果和决策
- **technical-director**：可实现性问题和性能考虑
- **producer**：平衡风险和资源影响

### 协调对象

- **game-designer**：讨论设计意图和修订建议
- **gameplay-programmer**：验证可实现性评估
- **qa-lead**：确认验收标准的可测试性

## 常见错误

### 错误1：过于宽松的审查

**问题**：因为"设计者知道意思"而忽略缺失的细节

**修正**：程序员不是设计者，必须按字面理解。缺失细节必须明确指出

### 错误2：混淆设计意图与实现细节

**问题**：在审查中要求具体的实现方法（如"必须使用哈希表"）

**修正**：设计审查关注"做什么"，实现细节留给程序员决定

### 错误3：忽略跨系统一致性

**问题**：只检查文档内部，不检查与现有系统的冲突

**修正**：必须读取相关系统文档，检查冲突和意外交互

### 错误4：决策不明确

**问题**：输出"需要一些修改"而非明确的"批准/需要修订/需要重大修订"

**修正**：必须做出明确决策，使用标准决策术语
