# Design Review

审查游戏设计文档的完整性、内部一致性、可实现性和对项目设计标准的遵循。在将设计文档交给程序员之前运行此审查。

## 审查要素

### 1. 完整性检查 (Completeness Check)
- 概述部分：一段摘要
- 玩家幻想部分：预期感受
- 详细规则部分：明确的机制
- 公式部分：所有数学定义带变量
- 边缘情况部分：处理异常情况
- 依赖关系部分：列出其他系统
- 调整参数部分：识别可配置值
- 验收标准部分：可测试的成功条件

### 2. 内部一致性 (Internal Consistency)
- 公式一致性：公式产生与描述行为匹配的值
- 边缘情况一致性：边缘情况不与主规则矛盾
- 依赖关系一致性：依赖关系是双向的

### 3. 可实现性 (Implementability)
- 规则精确性：规则足够精确，程序员无需猜测即可实现
- 细节完整性：没有缺失细节的"挥手"部分
- 性能考虑：考虑性能影响

### 4. 跨系统一致性 (Cross-System Consistency)
- 冲突检查：是否与现有机制冲突
- 交互检查：是否与其他系统产生意外交互
- 基调一致性：与游戏既定基调和支柱一致

## 审查原则

### 1. 全面性
- 全面审查所有部分
- 全面检查一致性
- 全面检查可实现性

### 2. 精确性
- 规则必须精确
- 公式必须明确
- 细节必须完整

### 3. 可操作性
- 提供可操作建议
- 提供优先改进列表
- 提供明确决策

### 4. 及时性
- 及时审查
- 及时反馈
- 及时修正

## 审查流程

### 1. 读取目标设计文档
- 完整读取目标设计文档
- 理解设计内容
- 理解设计目标

### 2. 读取主CLAUDE.md
- 理解项目上下文
- 理解项目标准
- 理解项目约束

### 3. 读取相关设计文档
- 检查 design/gdd/ 中的相关系统
- 理解相关设计
- 理解依赖关系

### 4. 评估设计文档标准
- 检查概述部分
- 检查玩家幻想部分
- 检查详细规则部分
- 检查公式部分
- 检查边缘情况部分
- 检查依赖关系部分
- 检查调整参数部分
- 检查验收标准部分

### 5. 检查内部一致性
- 检查公式一致性
- 检查边缘情况一致性
- 检查依赖关系一致性

### 6. 检查可实现性
- 检查规则精确性
- 检查细节完整性
- 检查性能考虑

### 7. 检查跨系统一致性
- 检查冲突
- 检查意外交互
- 检查基调一致性

### 8. 输出审查
- 输出完整性评估
- 输出一致性问题
- 输出可实现性问题
- 输出平衡问题
- 输出建议
- 输出决策

### 9. 上下文后续步骤建议
- 如果审查的是 game-concept.md 或 game-pillars.md
- 如果审查的是单个系统GDD
- 提供后续步骤建议
