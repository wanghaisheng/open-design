# Level Design

设计游戏关卡和遭遇，创建引人入胜的空间设计和玩家体验。

## 适用场景

- 从 0 到 1 设计一个可落地实现的单关/章节关卡（线性/半开放/开放）
- 需要把“关卡体验”拆成空间、遭遇、节奏与引导，并明确验收标准
- 关卡出现卡关、迷路、节奏拖沓、资源分布不合理等问题，需要迭代优化
- 需要与关卡实现/美术/程序协作，约定触发器、刷怪点、资源点与性能预算

## 操作步骤

### 步骤1：写清关卡目标、约束与验收标准

**目标**：让关卡设计从一开始就可验收、可落地

**操作方法**：
1. 用 1-2 句写关卡目标（玩家在这一关要学会/体验/完成什么）
2. 列出约束（时长、难度区间、可用资源、引擎/平台、性能预算）
3. 写 3-5 条验收标准（例如：平均通关 8-10 分钟；关键目标不需要外部攻略）
4. 写 3-5 条失败信号（例如：某处死亡率过高；迷路率过高；离开率异常）

**检查清单**：
- [ ] 目标明确且可测
- [ ] 约束明确（时长/难度/资源/预算）
- [ ] 有验收标准与失败信号

### 步骤2：搭关卡骨架（主路径 + 分支 + 回路）

**目标**：先把空间流线跑通，避免后期推倒重来

**操作方法**：
1. 画主路径：起点-关键节点-终点
2. 标注分支：奖励支线/风险支线/探索支线（每条都写收益与代价）
3. 标注回路：让玩家能回到主路径，避免死路与迷路
4. 为关键节点写“玩家状态”假设（资源、血量、技能、装备）

**检查清单**：
- [ ] 主路径可一口气跑通
- [ ] 分支有明确收益与回主路径方式
- [ ] 关键节点有玩家状态假设

### 步骤3：设计遭遇（战斗/谜题/社交）并绑定触发条件

**目标**：把遭遇从“点子”写成可实现的触发与结果

**操作方法**：
1. 为每个遭遇写目的（教学/考核/奖励/叙事）
2. 写触发条件（进入区域/拾取物/击败某敌/时间到）
3. 写胜负条件与失败反馈（重试点、提示策略、奖励扣减）
4. 写资源投放与回收（弹药/血包/金币/关键道具）

**检查清单**：
- [ ] 每个遭遇都有目的与触发条件
- [ ] 有胜负条件与失败反馈
- [ ] 资源投放与回收可解释

### 步骤4：编排节奏（紧张-放松-高潮）并设置检查点

**目标**：控制玩家的负荷与休息点，稳定体验曲线

**操作方法**：
1. 把关卡拆成 3-7 个节奏段（每段写紧张度与目标）
2. 在高强度段前后安排缓冲（补给、叙事、探索）
3. 设置检查点（复活点/存档点）与回滚策略
4. 标注“高潮段”与“收束段”的信息密度与操作密度

**检查清单**：
- [ ] 节奏段清晰
- [ ] 有缓冲与补给
- [ ] 检查点位置合理（不惩罚、不刷屏）

### 步骤5：设计引导（视觉/环境/教学）并定义失败兜底

**目标**：让玩家“不看攻略也能推进”，减少迷路与挫败

**操作方法**：
1. 为每个关键目标设计 2-3 种引导信号（灯光/地标/动线/音效/镜头）
2. 为新机制设计教学：提示-演示-练习-考核（最少一次闭环）
3. 定义失败兜底策略（提示触发条件、重复次数、可跳过机制）
4. 检查可访问性（色弱、对比度、字幕、UI提示）

**检查清单**：
- [ ] 关键目标有冗余引导
- [ ] 教学闭环完整
- [ ] 有失败兜底策略

### 步骤6：输出实现清单并组织测试迭代

**目标**：让关卡实现团队可直接落地，并能高效迭代

**操作方法**：
1. 输出关卡对象清单（触发器、刷怪点、门、机关、检查点、收集物）
2. 输出参数表（敌人数量、刷新规则、掉落、检查点规则）
3. 输出测试用例（通关/边界/异常/性能）
4. 收集数据与反馈（死亡点、停留时长、迷路点、跳出点）并形成迭代记录

**检查清单**：
- [ ] 实现清单完整且可分工
- [ ] 参数可配置且有默认值/范围
- [ ] 测试用例可复现
- [ ] 有迭代记录（问题-假设-改动-结果）

## 操作注意事项

### 清晰性与可读性

1. 目标必须“可见”：玩家在 10 秒内能知道自己该做什么。
2. 路径必须“可验证”：关键转折处提供地标或强提示。

### 多样性与重玩

1. 分支要有真实决策成本，不要只有“顺路捡奖励”。
2. 同类遭遇避免重复模板刷屏（换规则、换空间、换资源压力）。

### 平衡与容错

1. 卡点优先用“提示/兜底”解决，其次才是降难度。
2. 检查点不要让玩家重复做大量无意义操作。

## 输出格式

### 关卡设计文档

- 关卡目标、约束、验收标准、失败信号
- 关卡骨架图（主路径/分支/回路/关键节点）
- 遭遇清单（目的、触发、胜负条件、资源投放）
- 节奏编排（分段、缓冲、高潮、检查点）
- 引导方案（视觉/环境/教学 + 失败兜底）
- 实现清单（触发器/刷怪点/机关/检查点/收集物）
- 参数表（默认值/范围）
- 测试用例与迭代记录

## 协作协议

### 向谁汇报

- lead-programmer：触发器/刷怪/检查点可实现性与数据结构
- technical-director：性能预算、AI/寻路约束、系统耦合风险

### 协调对象

- gameplay-programmer：遭遇逻辑与教学闭环
- level-implementation：关卡搭建与触发器落地
- environment-artist：地标、引导视觉与性能优化
- qa-tester：卡点复现、通关路径与回归测试

## 常见错误

### 错误1：只写“主题/氛围”，没有路径骨架与触发条件

**问题**：实现阶段无法拆分任务，关卡容易推倒重来。

**修正**：按步骤2/3先补齐主路径骨架与遭遇触发条件。

### 错误2：缺少失败兜底，导致卡关靠试错

**问题**：玩家体验挫败，流失率上升。

**修正**：在步骤5补齐提示触发条件与可跳过/可恢复机制。

### 错误3：检查点设置不合理导致重复劳动

**问题**：玩家需要反复跑图或重复无意义操作。

**修正**：在步骤4重新规划检查点，并与节奏段对齐。
