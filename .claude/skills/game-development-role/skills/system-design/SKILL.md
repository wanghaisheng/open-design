# System Design

设计游戏系统，定义游戏机制、规则和交互，确保系统之间的协调和平衡。

## 适用场景

- 从 0 到 1 设计一个“可实现、可调参、可测试”的新系统（战斗/经济/成长/掉落/任务等）
- 需要把一个系统拆解为机制/规则/交互/平衡点，形成可落地的设计稿
- 现有系统上线后出现数值失衡、策略单一、体验割裂，需要定位原因并迭代
- 需要与程序、关卡、UI、数据分析协作，明确接口、数据、验收方式

## 操作步骤

### 步骤1：把系统边界与目标写成“可验收”的一句话

**目标**：明确系统解决什么问题、服务哪个体验目标、成功标准是什么

**操作方法**：
1. 写系统名称与一句话目标（面向玩家体验）
2. 列出系统输入/输出（玩家行为、资源、状态变化）
3. 写 3-5 条成功标准（可观测、可测试）
4. 写 3-5 条失败信号（出现即视为需要调整）

**检查清单**：
- [ ] 有清晰边界：系统“包含什么/不包含什么”
- [ ] 有成功标准：能用数据或可重复测试验证
- [ ] 有失败信号：能快速触发回滚或热修策略

### 步骤2：用“机制/规则/交互/平衡”四块拆解系统

**目标**：把系统拆成可实现的设计单元，避免只写概念

**操作方法**：
1. **机制（Mechanics）**：列核心机制、次级机制、元机制（每条都写触发条件与结果）
2. **规则（Rules）**：列明确规则、隐含规则、玩家规则（每条规则写允许/禁止/例外）
3. **交互（Interaction）**：列玩家-系统、玩家-玩家、系统-系统交互（写输入方式与反馈）
4. **平衡点（Balance）**：列关键参数与目标区间（数值/策略/时间），标注风险

**检查清单**：
- [ ] 每条机制/规则都可被程序实现（有触发与结果）
- [ ] 每个交互都有反馈（提示/动画/音效/数值变更）
- [ ] 每个平衡点都有“目标区间”与“测试方法”

### 步骤3：定义数据与接口（让程序能接）

**目标**：把“设计”落到数据结构与接口约定，降低返工

**操作方法**：
1. 列系统核心数据表（字段、类型、含义、默认值、范围）
2. 列系统关键事件/触发（何时触发、携带数据、影响对象）
3. 列对外接口（被谁调用、调用时机、输入输出）
4. 写最小集成路径（先接哪里、如何灰度/开关）

**检查清单**：
- [ ] 字段可配置且有范围
- [ ] 接口有调用时机与错误处理约定
- [ ] 有灰度/开关策略（便于迭代与回滚）

### 步骤4：制定“最小可用”版本与验证方案

**目标**：先做可跑通的 MVP 版本，再迭代深度与平衡

**操作方法**：
1. 选出 MVP 机制与规则（最多 3-5 个关键点）
2. 定义 MVP 的可玩闭环（进入-消耗-产出-反馈-退出）
3. 设计测试用例（功能正确、边界条件、异常路径）
4. 设计平衡测试（参数扫描、对比组、极端值）

**检查清单**：
- [ ] MVP 可独立运行并可验收
- [ ] 有边界/异常测试用例
- [ ] 有至少一种平衡测试方式（参数扫描或对比组）

### 步骤5：迭代与调参（形成可追踪的变更记录）

**目标**：用可追踪的方式迭代，避免“凭感觉改”导致回退困难

**操作方法**：
1. 每次只改 1-3 个参数/规则，并记录假设
2. 对比改动前后：关键指标、玩家反馈、复现步骤
3. 固化结论：哪些参数有效、哪些无效、下一步怎么试

**检查清单**：
- [ ] 变更有记录（改了什么、为什么、结果如何）
- [ ] 能复现（测试步骤与版本号明确）
- [ ] 有下一步计划（继续试/回滚/扩大灰度）

## 操作注意事项

### 可实现性

1. 避免把系统写成口号；每条机制/规则都要能映射到“触发条件 + 结果”。
2. 先明确数据与接口，再谈完整体验；否则程序落地时会被迫重写。

### 一致性与可理解

1. 命名要一致：同一资源/状态不要出现多个同义词。
2. 反馈要明确：玩家必须知道自己“做了什么、得到了什么、为什么”。

### 平衡与可调参

1. 参数必须有范围与默认值，并且能热更新（如果项目支持）。
2. 对“极端玩法”提前写失败信号与保护机制（上限、冷却、衰减）。

## 输出格式

### 系统设计文档

- 系统名称与一句话目标
- 系统边界（包含/不包含）
- 成功标准与失败信号
- 拆解：机制/规则/交互/平衡点（含目标区间）
- 数据表与字段定义（范围/默认值）
- 事件/接口约定（调用时机/输入输出/错误处理）
- MVP 范围与验收标准
- 测试用例与平衡测试方案
- 迭代记录（假设-改动-结果-结论）

## 协作协议

### 向谁汇报

- lead-programmer：接口可实现性、数据落地方式
- technical-director：跨系统影响、技术风险与架构约束

### 协调对象

- gameplay-programmer：机制实现与事件接口
- ui-programmer：反馈呈现与交互路径
- level-designer：系统在关卡中的触发点与节奏
- analytics-engineer：指标与埋点设计

## 常见错误

### 错误1：只写“原则/愿景”，没有可实现的触发与结果

**问题**：文档看起来正确，但程序无法实现或实现后与预期偏差巨大。

**修正**：按步骤2把每条内容写成“触发条件 + 结果 + 反馈”，并补齐数据与接口。

### 错误2：没有定义目标区间与测试方法，导致调参无从下手

**问题**：上线后只能靠感觉改，容易反复震荡。

**修正**：为关键参数写目标区间、失败信号，并在步骤4建立参数扫描/对比组测试。

### 错误3：系统边界不清，和其他系统互相污染

**问题**：改一个系统牵一发而动全身，迭代成本快速上升。

**修正**：在步骤1写清包含/不包含，并在步骤3明确对外接口与事件。
