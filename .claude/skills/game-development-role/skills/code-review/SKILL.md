# Code Review

对指定文件或文件集执行架构和质量代码审查。检查编码标准合规性、架构模式遵循性、SOLID原则、可测试性和性能问题。

## 适用场景

- 代码提交前的质量检查
- 需要检查编码标准合规性
- 需要检查架构模式遵循性
- 需要检查SOLID原则
- 需要检查游戏开发特定问题

## 操作步骤

### 步骤1：读取目标文件

**目标**：理解代码结构和目的

**操作方法**：
1. 完整读取目标文件
2. 理解代码结构
3. 理解代码目的

**检查清单**：
- [ ] 目标文件已读取
- [ ] 代码结构已理解
- [ ] 代码目的已理解

### 步骤2：读取编码标准

**目标**：理解项目编码标准和架构模式

**操作方法**：
1. 读取 CLAUDE.md
2. 读取 .claude/docs/technical-preferences.md
3. 理解项目编码标准
4. 理解架构模式

**检查清单**：
- [ ] 编码标准已读取
- [ ] 架构模式已理解

### 步骤3：识别系统类别

**目标**：应用类别特定标准

**操作方法**：
1. 识别系统类别（引擎、游戏玩法、AI、网络、UI、工具）
2. 应用类别特定标准

**检查清单**：
- [ ] 系统类别已识别
- [ ] 类别特定标准已应用

### 步骤4：评估编码标准合规性

**目标**：检查编码标准合规性

**操作方法**：
1. **文档注释**：检查公共方法和类有文档注释
2. **圈复杂度**：检查每个方法圈复杂度低于10
3. **方法长度**：检查没有方法超过40行（不包括数据声明）
4. **依赖注入**：检查依赖被注入（没有游戏状态的静态单例）
5. **配置值**：检查配置值从数据文件加载
6. **系统接口**：检查系统暴露接口（不依赖具体类）

**检查清单**：
- [ ] 文档注释已检查
- [ ] 圈复杂度已检查
- [ ] 方法长度已检查
- [ ] 依赖注入已检查
- [ ] 配置值已检查
- [ ] 系统接口已检查

### 步骤5：检查架构合规性

**目标**：检查架构模式遵循性

**操作方法**：
1. **依赖方向**：检查正确的依赖方向（引擎 <- 游戏玩法，而非反向）
2. **循环依赖**：检查模块间没有循环依赖
3. **层分离**：检查适当的层分离（UI不拥有游戏状态）
4. **事件/信号**：检查使用事件/信号进行跨系统通信
5. **模式一致性**：检查与代码库中已建立的模式一致

**检查清单**：
- [ ] 依赖方向已检查
- [ ] 循环依赖已检查
- [ ] 层分离已检查
- [ ] 事件/信号已检查
- [ ] 模式一致性已检查

### 步骤6：检查SOLID合规性

**目标**：检查SOLID原则

**操作方法**：
1. **单一职责**：检查每个类只有一个变更原因
2. **开闭原则**：检查可扩展无需修改
3. **里氏替换**：检查子类型可替换基类型
4. **接口隔离**：检查没有臃肿接口
5. **依赖倒置**：检查依赖抽象而非具体

**检查清单**：
- [ ] 单一职责已检查
- [ ] 开闭原则已检查
- [ ] 里氏替换已检查
- [ ] 接口隔离已检查
- [ ] 依赖倒置已检查

### 步骤7：检查游戏开发特定问题

**目标**：检查游戏开发常见问题

**操作方法**：
1. **帧率独立性**：检查帧率独立性（delta time使用）
2. **热路径分配**：检查热路径（更新循环）中没有分配
3. **空状态处理**：检查适当的空/空状态处理
4. **线程安全**：检查需要时线程安全
5. **资源清理**：检查资源清理（无泄漏）

**检查清单**：
- [ ] 帧率独立性已检查
- [ ] 热路径分配已检查
- [ ] 空状态处理已检查
- [ ] 线程安全已检查
- [ ] 资源清理已检查

### 步骤8：输出审查报告

**目标**：生成结构化审查报告

**操作方法**：
1. 输出标准合规性（X/6通过）
2. 输出架构状态（干净/轻微问题/发现违规）
3. 输出SOLID状态（合规/发现问题）
4. 输出游戏特定问题
5. 输出正面观察（做得好的地方）
6. 输出必需更改（批准前必须修复的项目）
7. 输出建议（有则更好的改进）
8. 输出结论（批准/批准带建议/需要更改）

**检查清单**：
- [ ] 标准合规性已输出
- [ ] 架构状态已输出
- [ ] SOLID状态已输出
- [ ] 游戏特定问题已输出
- [ ] 正面观察已输出
- [ ] 必需更改已输出
- [ ] 建议已输出
- [ ] 结论已输出

## 操作注意事项

### 审查的全面性

1. **全面审查代码**：不只看功能，要看所有方面（标准、架构、SOLID、游戏特定）
2. **检查所有方面**：编码标准、架构合规性、SOLID原则、游戏开发问题
3. **避免遗漏**：使用检查清单确保不遗漏任何检查项

### 建设性反馈

1. **提供建设性反馈**：不只指出问题，要提供改进建议
2. **提供具体建议**：建议应具体可操作
3. **避免批评**：关注代码而非个人

### 可操作的结果

1. **审查结果可操作**：每个问题都应有具体修复建议
2. **提供具体建议**：不只说"需要改进"，要说明如何改进
3. **指导后续工作**：明确哪些是必需更改，哪些是建议

### 正面观察

1. **始终包括正面观察**：不只指出问题，也要认可做得好的地方
2. **鼓励良好实践**：认可良好实践，鼓励团队学习

## 协作协议

### 向谁汇报

- **technical-lead**：代码审查结果和批准决定
- **technical-director**：架构问题和SOLID违规
- **qa-lead**：可测试性和质量问题

### 协调对象

- **gameplay-programmer**：编码标准和游戏特定问题
- **engine-programmer**：架构合规性和性能问题
- **ui-programmer**：UI层分离和事件系统

## 常见错误

### 错误1：只指出问题不提供解决方案

**问题**：只说"圈复杂度过高"而不说明如何降低

**修正**：必须提供具体修复建议，如"将此方法拆分为3个更小的方法"

### 错误2：忽略正面观察

**问题**：只指出问题，不认可做得好的地方

**修正**：必须始终包括正面观察部分，认可良好实践

### 错误3：结论不明确

**问题**：输出"需要一些修改"而非明确的"批准/批准带建议/需要更改"

**修正**：必须使用标准结论术语，明确说明批准状态

### 错误4：忽略游戏特定问题

**问题**：只检查通用编码标准，不检查游戏开发特定问题

**修正**：必须检查游戏开发特定问题（帧率独立性、热路径分配等）

### 错误5：忽略架构模式

**问题**：只检查编码标准，不检查架构模式遵循性

**修正**：必须检查架构合规性（依赖方向、循环依赖、层分离等）
