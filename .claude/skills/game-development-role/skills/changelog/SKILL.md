# Changelog

从git提交、冲刺数据和设计文档自动生成变更日志。生成内部版本和面向玩家的版本。

## 适用场景

- 需要生成版本变更日志
- 需要生成冲刺变更日志
- 需要生成内部技术变更日志
- 需要生成面向玩家的发布说明
- 需要跟踪版本历史

## 操作步骤

### 步骤1：读取参数

**目标**：确定变更日志的范围

**操作方法**：
1. 读取目标版本或冲刺编号
2. 使用相应的git标签或冲刺日期范围
3. 确定时间范围（开始日期、结束日期）

**检查清单**：
- [ ] 参数已读取
- [ ] 时间范围已确定

### 步骤2：检查git可用性

**目标**：验证git仓库可用

**操作方法**：
1. 验证仓库已初始化
2. 如果不是git仓库，通知用户

**检查清单**：
- [ ] git仓库已验证

### 步骤3：读取git日志

**目标**：获取git提交历史

**操作方法**：
1. 读取自上次标签或发布以来的日志
2. 如果没有标签，读取完整日志或合理范围
3. 记录提交哈希范围

**检查清单**：
- [ ] git日志已读取
- [ ] 提交哈希范围已记录

### 步骤4：读取冲刺报告

**目标**：理解计划工作和变更上下文

**操作方法**：
1. 读取 production/sprints/
2. 理解计划工作
3. 理解变更上下文

**检查清单**：
- [ ] 冲刺报告已读取
- [ ] 计划工作已理解
- [ ] 变更上下文已理解

### 步骤5：读取完成的设计文档

**目标**：识别期间实现的新功能

**操作方法**：
1. 读取 design/gdd/
2. 识别期间实现的任何新功能
3. 记录设计文档链接

**检查清单**：
- [ ] 设计文档已读取
- [ ] 新功能已识别
- [ ] 设计文档链接已记录

### 步骤6：分类每个变更

**目标**：将变更分类到标准类别

**操作方法**：
1. **新功能**：全新的游戏玩法系统、模式或内容
2. **改进**：现有功能的增强、UX改进、性能提升
3. **错误修复**：对错误行为的修正
4. **平衡变更**：游戏玩法值、难度、经济的调整
5. **已知问题**：团队已知但尚未解决的问题

**检查清单**：
- [ ] 变更已分类
- [ ] 类别已验证

### 步骤7：生成内部变更日志

**目标**：生成技术详细的内部变更日志

**操作方法**：
1. 包含完整技术细节
2. 记录受影响系统
3. 记录提交哈希
4. 记录负责人
5. 包含设计文档链接

**检查清单**：
- [ ] 技术细节已包含
- [ ] 受影响系统已记录
- [ ] 提交哈希已记录
- [ ] 负责人已记录
- [ ] 设计文档链接已包含

### 步骤8：生成面向玩家变更日志

**目标**：生成玩家友好的变更日志

**操作方法**：
1. 使用友好、非技术性语言
2. 描述玩家体验（而非实现）
3. 避免技术术语
4. 保持透明度

**检查清单**：
- [ ] 语言已友好化
- [ ] 玩家体验已描述
- [ ] 技术术语已避免
- [ ] 透明度已保持

### 步骤9：输出两个变更日志

**目标**：输出内部和面向玩家版本

**操作方法**：
1. 输出内部变更日志（主要工作文档）
2. 输出面向玩家变更日志（供审查后社区发布）

**检查清单**：
- [ ] 内部变更日志已输出
- [ ] 面向玩家变更日志已输出

## 输出格式

### 内部变更日志
```markdown
# 内部变更日志：[版本]
日期：[日期]
冲刺：[涵盖的冲刺编号]
提交：[计数]（[第一个哈希]..[最后一个哈希]）

## 新功能
- [功能名称] -- [技术描述，受影响系统]
  - 提交：[hash1], [hash2]
  - 负责人：[实现者]
  - 设计文档：[链接（如适用）]

## 改进
- [改进] -- [技术上改变了什么以及为什么]
  - 提交：[哈希]
  - 负责人：[谁]

## 错误修复
- [BUG-ID] [错误描述和根本原因]
  - 修复：[改变了什么]
  - 提交：[哈希]
  - 负责人：[谁]

## 平衡变更
- [调整了什么] -- [旧值 -> 新值] -- [设计意图]
  - 负责人：[谁]

## 技术债务/重构
- [清理了什么以及为什么]
  - 提交：[哈希]

## 已知问题
- [问题描述] -- [严重性] -- [修复预计时间（如已知）]

## 指标
- 总提交：[N]
- 更改文件：[N]
- 添加行：[N]
- 删除行：[N]
```

### 面向玩家变更日志
```markdown
# [版本] 的新内容

## 新功能
- **[功能名称]**：[玩家友好的描述，说明他们现在可以做什么以及为什么令人兴奋。专注于体验，而非实现。]

## 改进
- **[改进内容]**：[这如何让游戏对玩家更好。具体但避免术语。]

## 错误修复
- 修复了一个问题，[描述玩家经历了什么，而不是代码中有什么错误]
- 修复了[玩家可见的症状]

## 平衡变更
- [以玩家可理解的术语描述变化和设计意图。例如："治疗药水现在恢复50HP（从30提高）——我们觉得玩家在后期遭遇中需要更多恢复选项。"]

## 已知问题
- 我们意识到[玩家术语的问题描述]并正在修复。[如存在变通方法。]

---
感谢游玩！您的反馈帮助我们使游戏更好。
在[链接]报告问题。
```

## 规则

- 面向玩家变更日志中绝不暴露内部代码引用、文件路径或开发者姓名
- 将相关变更分组而非列出单个提交
- 如果提交消息不清楚，检查相关文件和冲刺数据以获取上下文
- 平衡变更应始终包含设计推理，不仅仅是数字
- 已知问题应诚实——玩家欣赏透明度
- 如果git历史混乱（合并提交、还原、修复提交），清理叙述而非字面列出每个提交

## 操作注意事项

### 准确性

1. **基于实际变更**：所有条目必须基于实际变更
2. **避免虚假条目**：不添加未实际发生的变更
3. **验证变更**：验证变更的真实性

### 完整性

1. **覆盖所有变更**：覆盖所有重要变更
2. **不遗漏重要变更**：不遗漏关键变更
3. **提供上下文**：为变更提供上下文

### 友好性

1. **面向玩家版本友好**：面向玩家版本使用友好语言
2. **避免技术术语**：避免技术术语
3. **专注体验**：专注玩家体验而非实现

### 透明度

1. **诚实面对已知问题**：诚实面对已知问题
2. **提供变通方法**：提供变通方法（如存在）
3. **解释变更原因**：解释变更原因

## 协作协议

### 向谁汇报

- **producer**：变更日志内容和发布计划
- **technical-lead**：内部变更日志的技术细节
- **community-manager**：面向玩家变更日志的审查

### 协调对象

- **scrum-master**：冲刺数据和变更上下文
- **game-designer**：设计文档和新功能描述
- **qa-lead**：错误修复和已知问题

## 常见错误

### 错误1：面向玩家版本暴露内部信息

**问题**：在面向玩家版本中暴露内部代码引用、文件路径或开发者姓名

**修正**：面向玩家变更日志中绝不暴露内部代码引用、文件路径或开发者姓名

### 错误2：列出单个提交而非分组

**问题**：逐个列出提交而非将相关变更分组

**修正**：将相关变更分组而非列出单个提交

### 错误3：提交消息不清楚时不检查上下文

**问题**：提交消息不清楚时不检查相关文件和冲刺数据

**修正**：如果提交消息不清楚，检查相关文件和冲刺数据以获取上下文

### 错误4：平衡变更只列数字不说明原因

**问题**：平衡变更只列数字变化，不说明设计推理

**修正**：平衡变更应始终包含设计推理，不仅仅是数字

### 错误5：不诚实面对已知问题

**问题**：隐藏已知问题或淡化问题严重性

**修正**：已知问题应诚实——玩家欣赏透明度
