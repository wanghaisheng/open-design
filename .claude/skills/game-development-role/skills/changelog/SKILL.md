# Changelog

从git提交、冲刺数据和设计文档自动生成变更日志。生成内部版本和面向玩家的版本。

## 变更日志要素

### 1. 变更分类
- 新功能：全新的游戏玩法系统、模式或内容
- 改进：现有功能的增强、UX改进、性能提升
- 错误修复：对错误行为的修正
- 平衡变更：游戏玩法值、难度、经济的调整
- 已知问题：团队已知但尚未解决的问题

### 2. 内部变更日志
- 技术细节
- 受影响系统
- 提交哈希
- 负责人
- 设计文档链接

### 3. 面向玩家变更日志
- 友好、非技术性
- 玩家体验描述
- 避免技术术语
- 透明度

### 传统工具
- 版本控制：Git

- 设计文档：GDD
- 问题跟踪：问题跟踪器

## 变更日志原则

### 1. 准确性
- 基于实际变更
- 避免虚假条目
- 验证变更

### 2. 完整性
- 覆盖所有变更
- 不遗漏重要变更
- 提供上下文

### 3. 友好性
- 面向玩家版本友好
- 避免技术术语
- 专注体验

### 4. 透明度
- 诚实面对已知问题
- 提供变通方法
- 解释变更原因

## 变更日志流程

### 1. 读取参数
- 目标版本或冲刺编号
- 使用相应的git标签或冲刺日期范围

### 2. 检查git可用性
- 验证仓库已初始化
- 如果不是git仓库，通知用户

### 3. 读取git日志
- 自上次标签或发布以来的日志
- 如果没有标签，读取完整日志或合理范围

### 4. 读取冲刺报告
- 读取 production/sprints/
- 理解计划工作和变更上下文

### 5. 读取完成的设计文档
- 读取 design/gdd/
- 识别期间实现的任何新功能

### 6. 分类每个变更
- 新功能
- 改进
- 错误修复
- 平衡变更
- 已知问题

### 7. 生成内部变更日志
- 完整技术细节
- 受影响系统
- 提交哈希
- 负责人

### 8. 生成面向玩家变更日志
- 友好、非技术性
- 玩家体验描述
- 避免技术术语

### 9. 输出两个变更日志
- 内部变更日志是主要工作文档
- 面向玩家变更日志准备好供审查后社区发布

## 输出格式

### 内部变更日志
```markdown
# 内部变更日志：[版本]
日期：[日期]
冲刺：[涵盖的冲刺编号]
提交：[计数]（[第一个哈希]..[最后一个哈希]）

## 新功能
- [功能名称] -- [技术描述，受影响系统]
  - 提交：[hash1], [hash2]
  - 负责人：[实现者]
  - 设计文档：[链接（如适用）]

## 改进
- [改进] -- [技术上改变了什么以及为什么]
  - 提交：[哈希]
  - 负责人：[谁]

## 错误修复
- [BUG-ID] [错误描述和根本原因]
  - 修复：[改变了什么]
  - 提交：[哈希]
  - 负责人：[谁]

## 平衡变更
- [调整了什么] -- [旧值 -> 新值] -- [设计意图]
  - 负责人：[谁]

## 技术债务/重构
- [清理了什么以及为什么]
  - 提交：[哈希]

## 已知问题
- [问题描述] -- [严重性] -- [修复预计时间（如已知）]

## 指标
- 总提交：[N]
- 更改文件：[N]
- 添加行：[N]
- 删除行：[N]
```

### 面向玩家变更日志
```markdown
# [版本] 的新内容

## 新功能
- **[功能名称]**：[玩家友好的描述，说明他们现在可以做什么以及为什么令人兴奋。专注于体验，而非实现。]

## 改进
- **[改进内容]**：[这如何让游戏对玩家更好。具体但避免术语。]

## 错误修复
- 修复了一个问题，[描述玩家经历了什么，而不是代码中有什么错误]
- 修复了[玩家可见的症状]

## 平衡变更
- [以玩家可理解的术语描述变化和设计意图。例如："治疗药水现在恢复50HP（从30提高）——我们觉得玩家在后期遭遇中需要更多恢复选项。"]

## 已知问题
- 我们意识到[玩家术语的问题描述]并正在修复。[如存在变通方法。]

---
感谢游玩！您的反馈帮助我们使游戏更好。
在[链接]报告问题。
```

## 规则

- 面向玩家变更日志中绝不暴露内部代码引用、文件路径或开发者姓名
- 将相关变更分组而非列出单个提交
- 如果提交消息不清楚，检查相关文件和冲刺数据以获取上下文
- 平衡变更应始终包含设计推理，不仅仅是数字
- 已知问题应诚实——玩家欣赏透明度
- 如果git历史混乱（合并提交、还原、修复提交），清理叙述而非字面列出每个提交
