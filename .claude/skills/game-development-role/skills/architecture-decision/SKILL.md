# Architecture Decision

创建架构决策记录（ADR），记录重要技术决策、上下文、考虑的替代方案和后果。每个主要技术选择都应该有ADR。

## 适用场景

- 需要记录重要技术决策
- 需要分析技术替代方案
- 需要评估决策后果
- 需要追踪架构决策历史
- 需要确保决策可追溯

## 操作步骤

### 步骤1：确定下一个ADR编号

**目标**：确定ADR编号

**操作方法**：
1. 扫描 docs/architecture/ 目录
2. 确定下一个ADR编号

**检查清单**：
- [ ] ADR编号已确定

### 步骤2：收集上下文

**目标**：收集决策上下文信息

**操作方法**：
1. 阅读相关代码
2. 阅读现有ADR
3. 收集技术约束（技术、时间、资源、兼容性）
4. 收集需求要求（功能、性能、集成）

**检查清单**：
- [ ] 相关代码已阅读
- [ ] 现有ADR已阅读
- [ ] 技术约束已收集
- [ ] 需求要求已收集

### 步骤3：引导决策

**目标**：通过澄清问题引导用户

**操作方法**：
1. 通过澄清问题引导用户
2. 分析技术选项
3. 评估替代方案

**检查清单**：
- [ ] 问题已澄清
- [ ] 技术选项已分析
- [ ] 替代方案已评估

### 步骤4：生成ADR

**目标**：使用ADR模板生成文档

**操作方法**：
1. 使用ADR模板
2. 填充上下文（问题陈述、约束条件、需求）
3. 填充决策（技术决策、架构图、关键接口）
4. 填充替代方案（描述、优点、缺点、拒绝原因）
5. 填充后果（正面影响、负面影响、风险）
6. 填充性能影响（CPU、内存、加载时间、网络）
7. 填充迁移计划
8. 填充验证标准
9. 填充相关决策

**检查清单**：
- [ ] 上下文已填充
- [ ] 决策已填充
- [ ] 替代方案已填充
- [ ] 后果已填充
- [ ] 性能影响已填充
- [ ] 迁移计划已填充
- [ ] 验证标准已填充
- [ ] 相关决策已填充

### 步骤5：保存ADR

**目标**：保存ADR到版本控制

**操作方法**：
1. 保存到 docs/architecture/adr-[NNNN]-[slug].md
2. 提交到版本控制
3. 通知相关团队

**检查清单**：
- [ ] ADR已保存
- [ ] 已提交到版本控制
- [ ] 相关团队已通知

## 常见错误

### 错误1：不收集上下文

**问题**：不阅读相关代码和现有ADR

**修正**：必须阅读相关代码和现有ADR，收集技术约束和需求要求

### 错误2：不评估替代方案

**问题**：只记录决策，不评估替代方案

**修正**：必须评估至少2-3个替代方案，说明拒绝原因

### 错误3：不定义验证标准

**问题**：不定义如何验证决策正确性

**修正**：必须定义验证标准、验证方法和验证时间

### 错误4：不考虑迁移计划

**问题**：不说明如何从现有代码迁移到新方案

**修正**：必须提供迁移计划（如果改变了现有代码）

### 错误5：不记录性能影响

**问题**：不评估决策对性能的影响

**修正**：必须评估性能影响（CPU、内存、加载时间、网络）
