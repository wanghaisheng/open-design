# Architecture Decision

创建架构决策记录（ADR），记录重要技术决策、上下文、考虑的替代方案和后果。每个主要技术选择都应该有ADR。

## ADR要素

### 1. 上下文 (Context)
- 问题陈述：我们要解决什么问题
- 约束条件：技术、时间、资源、兼容性约束
- 需求要求：必须支持的功能、性能预算、集成需求

### 2. 决策 (Decision)
- 技术决策：具体的技术决策
- 架构图：系统架构图
- 关键接口：API契约或接口定义

### 3. 替代方案 (Alternatives)
- 替代方案1：描述、优点、缺点、拒绝原因
- 替代方案2：描述、优点、缺点、拒绝原因
- 替代方案3：描述、优点、缺点、拒绝原因

### 4. 后果 (Consequences)
- 正面影响：好的结果
- 负面影响：权衡和成本
- 风险：可能出错的事情、缓解措施

## 决策原则

### 1. 充分分析
- 充分分析问题
- 充分分析替代方案
- 充分分析后果

### 2. 可追溯性
- 记录决策过程
- 记录决策原因
- 记录决策影响

### 3. 可验证性
- 定义验证标准
- 定义验证方法
- 定义验证时间

### 4. 可撤销性
- 考虑撤销成本
- 考虑迁移计划
- 考虑替代方案

## 决策流程

### 1. 确定下一个ADR编号
- 扫描docs/architecture/目录
- 确定下一个ADR编号

### 2. 收集上下文
- 阅读相关代码
- 阅读现有ADR
- 收集技术约束

### 3. 引导决策
- 通过澄清问题引导用户
- 分析技术选项
- 评估替代方案

### 4. 生成ADR
- 使用ADR模板
- 填充所有部分
- 确保完整性

### 5. 保存ADR
- 保存到docs/architecture/adr-[NNNN]-[slug].md
- 提交到版本控制
- 通知相关团队
