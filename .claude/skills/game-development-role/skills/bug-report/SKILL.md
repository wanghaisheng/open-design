# Bug Report

从描述创建结构化错误报告，或分析代码以识别潜在错误。确保每个错误报告都有完整的重现步骤、严重性评估和上下文。

## 错误报告要素

### 1. 摘要 (Summary)
- 标题：简洁、描述性标题
- ID：BUG-[NNNN]
- 严重性：S1-严重 / S2-主要 / S3-次要 / S4-琐碎
- 优先级：P1-立即 / P2-下一冲刺 / P3-待办事项 / P4-愿望清单
- 状态：开放
- 报告日期：报告日期
- 报告者：报告者姓名

### 2. 分类 (Classification)
- 类别：游戏玩法 / UI / 音频 / 视觉 / 性能 / 崩溃 / 网络
- 系统：受影响的游戏系统
- 频率：总是 / 经常（>50%）/ 有时（10-50%）/ 罕见（<10%）
- 回归：是/否/未知——之前是否工作正常？

### 3. 环境 (Environment)
- 版本：版本或提交哈希
- 平台：操作系统、硬件（如果相关）
- 场景/关卡：游戏中的位置
- 游戏状态：相关状态——物品栏、任务进度等

### 4. 重现步骤 (Reproduction Steps)
- 前置条件：开始前所需的状态
- 精确步骤：重现错误的精确步骤
- 预期结果：应该发生什么
- 实际结果：实际发生了什么

### 5. 技术上下文 (Technical Context)
- 可能受影响的文件：基于代码库搜索的文件列表
- 相关系统：可能涉及的其他系统
- 可能的根本原因：如果可从描述中识别

### 6. 证据 (Evidence)
- 日志：相关日志输出（如果可用）
- 视觉：视觉证据描述

## 错误分析要素

### 1. 潜在错误识别 (Potential Bug Identification)
- 空引用：空引用错误
- 差一错误：循环和数组索引的差一错误
- 竞争条件：竞争条件
- 未处理的边缘情况：未处理的边缘情况
- 资源泄漏：资源泄漏
- 错误的状态转换：错误的状态转换

### 2. 触发场景 (Trigger Scenario)
- 可能的触发场景：错误可能触发的场景
- 推荐修复：推荐的修复方法

## 错误报告原则

### 1. 完整性
- 完整的重现步骤
- 完整的环境信息
- 完整的技术上下文

### 2. 清晰性
- 清晰的描述
- 清晰的步骤
- 清晰的预期vs实际

### 3. 可操作性
- 可重现的错误
- 可修复的错误
- 可验证的修复

### 4. 优先级管理
- 正确的严重性评估
- 正确的优先级分配
- 正确的分类

## 错误报告流程

### 1. 解析描述
- 解析描述中的关键信息
- 识别错误类型
- 识别受影响的系统

### 2. 搜索代码库
- 使用Grep/Glob搜索相关文件
- 添加技术上下文
- 识别可能受影响的文件

### 3. 生成错误报告
- 填充摘要
- 填充分类
- 填充环境
- 填充重现步骤
- 填充技术上下文
- 填充证据

### 4. 分析代码（分析模式）
- 读取目标文件
- 识别潜在错误
- 为每个潜在错误生成错误报告
