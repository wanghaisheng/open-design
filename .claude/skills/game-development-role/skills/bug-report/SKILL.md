# Bug Report

从描述创建结构化错误报告，或分析代码以识别潜在错误。确保每个错误报告都有完整的重现步骤、严重性评估和上下文。

## 适用场景

- 从玩家/测试人员描述中创建结构化错误报告
- 分析代码以识别潜在错误（空引用、竞争条件、资源泄漏等）
- 需要评估错误严重性（S1-S4）和优先级（P1-P4）
- 需要为错误分配正确的分类（游戏玩法/UI/音频/视觉/性能/崩溃/网络）
- 需要与开发团队协作修复和验证错误

## 操作步骤

### 步骤1：收集错误基本信息

**目标**：获取创建错误报告所需的所有基本信息

**操作方法**：
1. 记录错误标题（简洁、描述性，20字以内）
2. 分配错误ID（BUG-[NNNN]格式）
3. 评估严重性：
   - S1-严重：崩溃/数据丢失/主要功能不可用
   - S2-主要：功能受损/有变通方案
   - S3-次要：小问题/轻微影响
   - S4-琐碎：拼写错误/视觉小问题
4. 评估优先级：
   - P1-立即：阻塞发布/影响所有用户
   - P2-下一冲刺：重要但非阻塞
   - P3-待办事项：已知问题/计划修复
   - P4-愿望清单：低影响/有时间再修
5. 记录报告日期和报告者

**检查清单**：
- [ ] 标题简洁且描述性
- [ ] ID格式正确（BUG-[NNNN]）
- [ ] 严重性评估有依据
- [ ] 优先级评估符合当前冲刺目标

### 步骤2：确定错误分类和环境

**目标**：为错误分配正确的分类和环境信息，便于路由和重现

**操作方法**：
1. 确定错误类别：
   - 游戏玩法：机制/规则/平衡问题
   - UI：界面/菜单/控件问题
   - 音频：音效/音乐/语音问题
   - 视觉：图形/动画/特效问题
   - 性能：帧率/加载/内存问题
   - 崩溃：程序终止/无响应
   - 网络：连接/同步/延迟问题
2. 识别受影响的游戏系统（战斗/经济/社交等）
3. 记录重现频率：总是/经常(>50%)/有时(10-50%)/罕见(<10%)
4. 判断是否为回归（之前版本是否正常）
5. 记录环境信息：
   - 版本号或提交哈希
   - 平台（OS/硬件）
   - 场景/关卡
   - 游戏状态（物品栏/任务进度等）

**检查清单**：
- [ ] 类别选择准确
- [ ] 受影响的系统已识别
- [ ] 频率评估基于实际测试
- [ ] 环境信息完整

### 步骤3：编写重现步骤

**目标**：让任何人都能按步骤重现错误

**操作方法**：
1. 列出前置条件（开始前需要什么状态）
2. 编写精确的步骤（每步一个动作，编号）
3. 描述预期结果（应该发生什么）
4. 描述实际结果（实际发生了什么，与预期差异）
5. 验证步骤：尝试按自己的步骤重现，确保无歧义

**检查清单**：
- [ ] 前置条件明确
- [ ] 步骤精确无歧义
- [ ] 预期结果清晰
- [ ] 实际结果描述准确
- [ ] 通过第三方验证（让其他人按步骤重现）

### 步骤4：收集技术上下文和证据

**目标**：为开发人员提供诊断和修复所需的技术信息

**操作方法**：
1. 使用搜索工具（Grep/Glob）查找可能相关的代码文件
2. 列出可能受影响的文件路径
3. 识别可能涉及的其他系统（跨系统交互）
4. 如果可从描述中识别，推测可能的根本原因
5. 收集证据：
   - 相关日志输出（错误日志/堆栈跟踪）
   - 截图/录屏（视觉问题）
   - 性能分析数据（性能问题）

**检查清单**：
- [ ] 相关代码文件已识别
- [ ] 涉及的其他系统已列出
- [ ] 根本原因有依据（如可识别）
- [ ] 日志/截图等证据已附加

### 步骤5：分析代码识别潜在错误（分析模式）

**目标**：主动识别代码中的潜在错误

**操作方法**：
1. 读取目标代码文件
2. 检查以下常见错误模式：
   - 空引用：未检查null就使用对象
   - 差一错误：循环和数组索引（i <= length 应为 i < length）
   - 竞争条件：多线程访问共享资源无同步
   - 未处理的边缘情况：空数组/零除/边界值
   - 资源泄漏：未释放的文件句柄/内存/连接
   - 错误的状态转换：无效的状态机转换
3. 对每个潜在错误生成错误报告（使用步骤1-4的格式）
4. 为每个潜在错误推荐修复方法

**检查清单**：
- [ ] 代码已完整阅读
- [ ] 常见错误模式已检查
- [ ] 每个潜在错误都有报告
- [ ] 修复建议具体可行

### 步骤6：输出结构化错误报告

**目标**：生成符合格式的错误报告文档

**操作方法**：
1. 使用标准错误报告模板
2. 填充所有字段（摘要、分类、环境、重现步骤、技术上下文、证据）
3. 添加相关问题链接（如果有）
4. 添加备注（额外上下文或观察）
5. 审核报告完整性

**检查清单**：
- [ ] 所有必填字段已填充
- [ ] 格式符合模板标准
- [ ] 无遗漏信息
- [ ] 已准备提交到错误跟踪系统

## 操作注意事项

### 严重性与优先级区分

1. **严重性** = 错误的影响程度（S1-S4），基于技术影响
2. **优先级** = 修复的紧急程度（P1-P4），基于业务目标
3. 高严重性不一定等于高优先级（例如：罕见崩溃但影响很小）
4. 低严重性可能等于高优先级（例如：发布前一天的拼写错误）

### 重现步骤的可验证性

1. 步骤必须具体到按钮名称、菜单路径
2. 避免模糊的描述（如"做一些事情"）
3. 如果错误是间歇性的，注明频率和条件
4. 让第三方按步骤验证，确认可重现

### 技术上下文的准确性

1. 不要猜测根本原因，除非有明确证据
2. 代码引用要精确到文件路径和行号
3. 日志摘要要相关，不要包含无关信息
4. 如果不确定涉及哪些系统，列出可能性而非断言

- network-programmer：网络错误的重现和修复
- qa-tester：验证修复和回归测试

## 常见错误

### 错误1：标题过于模糊

**问题**：标题如"游戏出错了"或"有问题"，无法快速理解错误本质

**修正**：标题应包含错误位置和现象，如"战斗系统：玩家攻击后角色消失"

### 错误2：重现步骤不完整

**问题**：缺少前置条件或关键步骤，导致其他人无法重现

**修正**：总是列出前置条件，每个步骤具体到按钮/菜单名称，验证第三方可重现

### 错误3：缺少实际结果描述

**问题**：只描述"出错了"，没有说明具体发生什么

**修正**：描述实际结果时应具体（如"角色生命值变为负数，UI显示NaN"）

### 错误4：混淆严重性与优先级

**问题**：将所有崩溃标记为P1，或将所有视觉问题标记为P4

**修正**：严重性基于技术影响，优先级基于业务目标。罕见崩溃可能是S1但P3，发布前一天的拼写错误可能是S4但P1

### 错误5：技术上下文过于推测

**问题**：猜测根本原因但无代码证据，误导开发人员

**修正**：如果没有明确证据，列出可能性并说明需要代码审查，不要断言根本原因
