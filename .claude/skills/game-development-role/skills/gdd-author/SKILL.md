# GDD Author

结构化和维护活的游戏设计文档——游戏设计的单一真相来源。

## 适用场景

- 需要从现有设计输出创建GDD
- 需要构建模块化可导航文档
- 需要确定不同受众的适当详细程度
- 需要集成变更跟踪
- 需要生成各领域部分模板

## 操作步骤

### 步骤1：收集技能输出

**目标**：收集所有技能输出

**操作方法**：
1. 收集愿景输出（vision技能）
2. 收集支柱输出（design-pillars技能）
3. 收集美学输出（aesthetics技能）
4. 收集体验输出（player-experience技能）
5. 收集循环输出（core-loop技能）
6. 收集规则输出（rules技能）
7. 收集系统输出（systems技能）
8. 收集经济输出（economy技能）
9. 收集平衡输出（balance技能）
10. 收集叙事输出（narrative技能）
11. 收集关卡输出（level-design技能）
12. 收集世界输出（world-design技能）
13. 收集UI输出（ui-design技能）
14. 收集范围输出（scope技能）
15. 收集技术输出（technical技能）

**检查清单**：
- [ ] 愿景输出已收集
- [ ] 支柱输出已收集
- [ ] 美学输出已收集
- [ ] 体验输出已收集
- [ ] 循环输出已收集
- [ ] 规则输出已收集
- [ ] 系统输出已收集
- [ ] 经济输出已收集
- [ ] 平衡输出已收集
- [ ] 叙事输出已收集
- [ ] 关卡输出已收集
- [ ] 世界输出已收集
- [ ] UI输出已收集
- [ ] 范围输出已收集
- [ ] 技术输出已收集

### 步骤2：选择GDD变体

**目标**：选择GDD变体

**操作方法**：
1. 评估项目规模（游戏开发、独立、发行商推介、完整生产）
2. 评估受众（内部团队、发行商、投资者）
3. 选择GDD变体（游戏开发GDD 1-2页、独立GDD 5-15页、发行商推介GDD 10-20页、完整生产GDD 20-50+页）

**检查清单**：
- [ ] 项目规模已评估
- [ ] 受众已评估
- [ ] GDD变体已选择

### 步骤3：编写摘要

**目标**：编写每个部分的摘要

**操作方法**：
1. 为每个部分编写2-3句摘要
2. 摘要应包含核心要点
3. 摘要应具体而非渴望

**检查清单**：
- [ ] 每个部分摘要已编写
- [ ] 摘要包含核心要点
- [ ] 摘要具体而非渴望

### 步骤4：填充详情

**目标**：填充每个部分的详情

**操作方法**：
1. 从技能输出扩展详情
2. 大量使用示例
3. 使用前定义术语
4. 为扫描阅读而写（使用标题、要点、粗体关键术语）

**检查清单**：
- [ ] 详情已从技能输出扩展
- [ ] 示例已使用
- [ ] 术语已定义
- [ ] 为扫描阅读而写

### 步骤5：添加交叉引用

**目标**：添加交叉引用

**操作方法**：
1. 链接相关部分
2. 建立部分间连接
3. 维护交叉引用系统

**检查清单**：
- [ ] 相关部分已链接
- [ ] 部分间连接已建立
- [ ] 交叉引用系统已维护

### 步骤6：标记开放问题

**目标**：标记开放问题

**操作方法**：
1. 识别尚未决定的内容
2. 标记开放问题
3. 跟踪开放问题解决

**检查清单**：
- [ ] 开放问题已识别
- [ ] 开放问题已标记
- [ ] 开放问题解决已跟踪

### 步骤7：链接附录

**目标**：链接附录

**操作方法**：
1. 指向详细文档以获取深度
2. 链接完整规则文档
3. 链接平衡电子表格
4. 链接关卡蓝图
5. 链接艺术/音频参考板

**检查清单**：
- [ ] 详细文档已链接
- [ ] 完整规则文档已链接
- [ ] 平衡电子表格已链接
- [ ] 关卡蓝图已链接
- [ ] 艺术/音频参考板已链接

### 步骤8：审查受众

**目标**：审查受众

**操作方法**：
1. 每个读者都能找到需要的内容？
2. 程序员需要系统规格
3. 发行商需要愿景和市场
4. 艺术家需要美学方向

**检查清单**：
- [ ] 程序员能找到系统规格
- [ ] 发行商能找到愿景和市场
- [ ] 艺术家能找到美学方向

### 步骤9：维护GDD

**目标**：维护GDD

**操作方法**：
1. 任何技能输出变化时，更新相应GDD部分
2. 标记变更日期并链接到迭代日志
3. 标记过时部分（活跃项目30+天未更新）
4. 定期审查部分间一致性

**检查清单**：
- [ ] 技能输出变化时GDD已更新
- [ ] 变更日期已标记
- [ ] 过时部分已标记
- [ ] 部分间一致性已审查

## 输出

此技能产生：
1. GDD结构——模块化、可导航、适合受众
2. 部分模板——每个领域一致格式
3. GDD变体选择——按项目需求缩放
4. 交叉引用系统——部分链接到相关部分
5. 维护协议——如何以及何时更新

这些输出集成：
- 所有技能——GDD综合每个活跃技能的输出
- 迭代技能——变更跟踪馈送GDD更新
- 一致性技能——GDD是设计一致性的可读表达

## 操作注意事项

### 活文档，非纪念碑

1. **GDD随设计变化而变化**：GDD随设计变化而变化
2. **完成的GDD意味着游戏已发布或取消**：完成的GDD意味着游戏已发布或取消

### 模块化，非整体

1. **没有人阅读200页文档**：没有人阅读200页文档
2. **GDD是可独立阅读的部分集合**：GDD是可独立阅读的部分集合

### 受众感知

1. **不同读者需要不同内容**：不同读者需要不同内容
2. **程序员需要系统规格**：程序员需要系统规格
3. **发行商需要愿景和市场**：发行商需要愿景和市场
4. **艺术家需要美学方向**：艺术家需要美学方向

### 真相来源，非一切来源

1. **GDD链接到详细文档**：GDD链接到详细文档
2. **不包含每个平衡电子表格和关卡蓝图**：不包含每个平衡电子表格和关卡蓝图
3. **提供确定性概述并指向深度**：提供确定性概述并指向深度

## 协作协议

### 向谁汇报

- **lead-designer**：GDD结构和内容
- **creative-director**：愿景和身份部分
- **producer**：范围和生产部分

### 协调对象

- **所有技能**：收集技能输出
- **design-iteration-tracker**：变更跟踪
- **consistency-check**：一致性验证

## 常见错误

### 错误1：不收集技能输出

**问题**：不收集所有技能输出，直接编写GDD

**修正**：必须收集所有技能输出（愿景、支柱、美学、体验、循环、规则、系统、经济、平衡、叙事、关卡、世界、UI、范围、技术）

### 错误2：不选择GDD变体

**问题**：不评估项目规模和受众，使用固定模板

**修正**：必须评估项目规模和受众，选择合适的GDD变体

### 错误3：不编写摘要

**问题**：不编写每个部分的摘要

**修正**：必须为每个部分编写2-3句摘要，包含核心要点

### 错误4：不添加交叉引用

**问题**：不添加交叉引用，部分间无连接

**修正**：必须添加交叉引用，链接相关部分

### 错误5：不维护GDD

**问题**：技能输出变化时不更新GDD

**修正**：必须维护GDD，技能输出变化时更新相应GDD部分
