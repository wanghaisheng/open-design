# GDD Author

结构化和维护活的游戏设计文档——游戏设计的单一真相来源。

## GDD 理念

### 活文档，非纪念碑
GDD 随设计变化而变化。"完成"的 GDD 意味着游戏已发布或取消。

### 模块化，非整体
没有人阅读200页文档。GDD 是可独立阅读的部分集合。

### 受众感知
不同读者需要不同内容。程序员需要系统规格。发行商需要愿景和市场。艺术家需要美学方向。为读者写作。

### 真相来源，非一切来源
GDD 链接到详细文档——不包含每个平衡电子表格和关卡蓝图。提供确定性概述并指向深度。

## GDD 结构

### 主模板

```
1. 概述
   1.1 愿景声明
   1.2 电梯推介
   1.3 核心幻想
   1.4 目标受众
   1.5 平台/媒介
   1.6 当前状态/版本

2. 身份
   2.1 设计支柱
   2.2 美学方向摘要
   2.3 对比框架和参考

3. 玩家体验
   3.1 情感弧线
   3.2 参与模型
   3.3 心流目标
   3.4 玩家动机档案

4. 核心游戏玩法
   4.1 核心循环图
   4.2 循环层次（微、核心、会话、元）
   4.3 关键机制摘要
   4.4 规则概述（摘要——完整规则链接）

5. 系统
   5.1 系统交互图
   5.2 经济和进度概述
   5.3 平衡框架

6. 内容
   6.1 叙事系统
   6.2 关卡/遭遇概述
   6.3 世界逻辑摘要

7. 界面
   7.1 UI/UX设计
   7.2 信息层次
   7.3 入门计划

8. 生产
   8.1 范围和优先级
   8.2 技术要求
   8.3 原型状态
   8.4 玩测历史
   8.5 迭代日志

9. 附录
   9.1 完整规则文档（链接）
   9.2 平衡电子表格（链接）
   9.3 关卡蓝图（链接）
   9.4 艺术/音频参考板（链接）
   9.5 词汇表
```

### 部分模板

每个部分遵循一致格式：

```
## [部分标题]

### 摘要
[2-3句话——核心要点]

### 当前状态
[已决定、进行中、开放的内容]

### 详情
[实质内容——因部分类型而异]

### 开放问题
[尚未决定的内容？]

### 变更历史
[此部分的最近变更，链接到迭代日志]

### 交叉引用
[连接到其他部分]
```

## GDD 变体

### 游戏开发GDD（1-2页）
- 愿景声明
- 核心循环图
- 3-5个支柱
- 关键机制列表
- 范围定义（包含什么、排除什么）

### 独立GDD（5-15页）
- 完整的概述和身份部分
- 核心游戏玩法部分
- 缩写的系统和内容
- 范围和优先级
- 无正式附录（工作文档作为参考）

### 发行商推介GDD（10-20页）
- 侧重概述、身份和玩家体验
- 市场定位和竞争分析
- 视觉/音频参考和概念艺术
- 生产时间表和预算概述
- 技术细节较少

### 完整生产GDD（20-50+页，模块化）
- 所有部分完全开发
- 链接到完整详细文档的附录
- 集成变更跟踪
- 多受众视图（设计视图、工程视图、艺术视图）

## 写作质量指南

### 具体而非渴望
"战斗系统使用2d6+修饰符解决和对抗掷骰"而非"战斗系统深入而引人入胜"。

### 大量使用示例
每个抽象概念都应有具体实例。"资源稀缺"变为"会话开始时，玩家有5金币。基础武器花费3。生命药水花费2。他们必须选择。"

### 使用前定义
每个游戏术语应首次出现时定义并列在词汇表中。

### 为扫描阅读而写
使用标题、要点、粗体关键术语和摘要。没有人线性阅读GDD——他们搜索需要的内容。

### 日期标注所有内容
每个部分应显示上次更新时间。过时部分应标记。

### 删除填充文本
如果部分没有具体内容（"经济将稳健和平衡"），删除直到有真实内容填充。

## 工作流程

### 从现有设计输出创建GDD

1. 收集所有技能输出——愿景、支柱、美学、体验、循环、规则、系统、经济、平衡、叙事、关卡、世界、UI、范围、技术
2. 选择GDD变体——匹配项目规模和受众
3. 先写摘要——每个部分以2-3句摘要开始
4. 填充详情——从技能输出扩展
5. 添加交叉引用——链接相关部分
6. 标记开放问题——尚未决定的内容？
7. 链接附录——指向详细文档以获取深度
8. 审查受众——每个读者都能找到需要的内容？

### 维护GDD

1. 任何技能输出变化时，更新相应GDD部分
2. 标记变更日期并链接到迭代日志
3. 标记过时部分（活跃项目30+天未更新）
4. 定期审查部分间一致性
5. GDD是设计负责人的责任，不是委员会

### 实践中使用GDD

- 团队对齐会议——参考特定部分
- 入门——新成员先阅读概述和身份，然后是其领域
- 决策制定——"这是否服务我们的支柱？"（部分2.1）
- 发行商沟通——提取推介GDD变体
- 范围讨论——参考部分8.1（范围和优先级）

## 反模式

### 博物馆藏品
编写一次永不更新的GDD。数周内变成虚构。

### 小说
太长无人阅读。GDD应可扫描，非阅读任务。

### 愿望清单
充满渴望功能，无约束、优先级或权衡。

### 委员会草案
共识编写无声音，充满规避语言和非承诺。

### 秘密文档
存在但无人能找到、访问或导航的GDD。

## 输出

此技能产生：
1. GDD结构——模块化、可导航、适合受众
2. 部分模板——每个领域一致格式
3. GDD变体选择——按项目需求缩放
4. 交叉引用系统——部分链接到相关部分
5. 维护协议——如何以及何时更新

这些输出集成：
- 所有技能——GDD综合每个活跃技能的输出
- 迭代技能——变更跟踪馈送GDD更新
- 一致性技能——GDD是设计一致性的可读表达
