# Milestone Review

生成全面的里程碑进度审查，包括功能完整性、质量指标、风险评估和通过/不通过建议。在里程碑检查点或评估里程碑截止日期准备度时使用。

## 适用场景

- 里程碑检查点的进度审查
- 评估里程碑截止日期准备度
- 需要生成里程碑通过/不通过建议
- 需要识别里程碑风险和阻塞
- 需要提供范围调整建议

## 操作步骤

### 步骤1：读取里程碑定义

**目标**：理解里程碑目标和范围

**操作方法**：
1. 读取 production/milestones/ 中的里程碑文档
2. 理解里程碑目标
3. 理解里程碑范围（功能列表）
4. 识别里程碑截止日期

**检查清单**：
- [ ] 里程碑文档已读取
- [ ] 里程碑目标已理解
- [ ] 里程碑范围已识别
- [ ] 截止日期已记录

### 步骤2：读取所有冲刺报告

**目标**：理解冲刺进度和速度趋势

**操作方法**：
1. 读取 production/sprints/ 中的所有冲刺报告
2. 理解冲刺进度（完成的任务/总任务）
3. 理解速度趋势（改善/稳定/下降）
4. 计算整体完成率

**检查清单**：
- [ ] 所有冲刺报告已读取
- [ ] 冲刺进度已理解
- [ ] 速度趋势已分析
- [ ] 整体完成率已计算

### 步骤3：评估功能完整性

**目标**：评估每个功能的完成状态

**操作方法**：
1. 比较里程碑范围与实际交付
2. 分类功能状态：
   - 完全完成：功能完全实现并通过验收
   - 部分完成：功能部分完成，剩余工作明确
   - 未开始：功能未开始，可考虑削减
3. 识别里程碑风险（未开始的关键功能）

**检查清单**：
- [ ] 功能状态已分类
- [ ] 里程碑风险已识别
- [ ] 削减候选已识别

### 步骤4：评估质量指标

**目标**：评估代码和产品质量

**操作方法**：
1. 检查错误跟踪器：
   - S1错误（严重）数量和列表
   - S2错误（重要）数量
   - S3错误（轻微）数量
2. 检查测试覆盖率
3. 检查性能是否在预算内（读取 technical-preferences.md）

**检查清单**：
- [ ] 错误统计已收集
- [ ] 测试覆盖率已检查
- [ ] 性能已评估

### 步骤5：扫描代码健康

**目标**：评估代码库健康状态

**操作方法**：
1. 使用 Grep 扫描代码库中的 TODO 标记
2. 扫描 FIXME 标记
3. 扫描 HACK 标记
4. 识别关键技术债务项目
5. 与之前里程碑比较，评估趋势

**检查清单**：
- [ ] TODO/FIXME/HACK 计数已统计
- [ ] 技术债务已识别
- [ ] 趋势已评估

### 步骤6：检查风险登记册

**目标**：评估风险状态和缓解措施

**操作方法**：
1. 读取 production/risk-register/
2. 评估每个风险的状态
3. 评估如果风险实现的影响
4. 评估缓解措施状态

**检查清单**：
- [ ] 风险登记册已读取
- [ ] 风险状态已评估
- [ ] 缓解措施已评估

### 步骤7：分析速度和调整估算

**目标**：基于当前速度调整剩余工作估算

**操作方法**：
1. 计算所有冲刺的计划vs完成
2. 识别速度趋势（改善/稳定/下降）
3. 基于当前速度计算剩余工作所需天数
4. 与剩余时间比较，评估可行性

**检查清单**：
- [ ] 速度已分析
- [ ] 趋势已识别
- [ ] 调整估算已计算
- [ ] 可行性已评估

### 步骤8：生成范围建议

**目标**：提供范围调整建议

**操作方法**：
1. **保护**：必须与里程碑一起交付的功能
2. **有风险**：可能需要削减或简化的功能
3. **削减候选**：可以推迟而不影响里程碑的功能
4. 说明每个建议的理由

**检查清单**：
- [ ] 保护功能已列出
- [ ] 有风险功能已列出
- [ ] 削减候选已列出
- [ ] 理由已说明

### 步骤9：生成通过/不通过评估

**目标**：做出明确的里程碑决策

**操作方法**：
1. 基于功能完整性、质量指标、风险、速度综合评估
2. 做出明确决策：
   - **通过**：所有目标达成，质量可接受
   - **条件通过**：有条件需满足（如修复关键错误）
   - **不通过**：无法达成目标，需要重大调整
3. 提供决策理由

**检查清单**：
- [ ] 综合评估已完成
- [ ] 决策已明确
- [ ] 理由已说明

### 步骤10：输出里程碑审查报告

**目标**：生成结构化审查报告

**操作方法**：
1. 输出概览（目标日期、当前日期、剩余天数）
2. 输出功能完整性（完全完成/部分完成/未开始）
3. 输出质量指标（错误、测试、性能）
4. 输出代码健康（TODO/FIXME/HACK）
5. 输出风险评估
6. 输出速度分析
7. 输出范围建议
8. 输出通过/不通过评估
9. 输出行动项

**检查清单**：
- [ ] 概览已输出
- [ ] 功能完整性已输出
- [ ] 质量指标已输出
- [ ] 代码健康已输出
- [ ] 风险评估已输出
- [ ] 速度分析已输出
- [ ] 范围建议已输出
- [ ] 决策已输出
- [ ] 行动项已输出

## 操作注意事项

### 审查的全面性

1. **全面审查功能**：不只看完成的任务，要看功能完整性
2. **全面审查质量**：不只看功能完成，要看质量指标
3. **全面审查风险**：不只看进度，要看风险状态

### 审查的客观性

1. **基于数据**：使用实际数据（错误统计、测试覆盖率、速度）
2. **避免主观判断**：不基于感觉，基于可验证指标
3. **可验证指标**：所有评估都应有数据支持

### 决策的明确性

1. **通过**：所有目标达成，质量可接受
2. **条件通过**：有条件需满足（如修复关键错误）
3. **不通过**：无法达成目标，需要重大调整

### 范围建议的谨慎性

1. **保护关键功能**：必须与里程碑一起交付的功能
2. **识别削减候选**：可以推迟而不影响里程碑的功能
3. **说明理由**：每个范围建议都有明确理由

## 协作协议

### 向谁汇报

- **producer**：里程碑审查结果和通过/不通过决策
- **technical-director**：代码健康和质量指标
- **qa-lead**：错误统计和测试覆盖率

### 协调对象

- **lead-designer**：功能完整性和范围建议
- **technical-lead**：技术债务和风险缓解
- **scrum-master**：速度分析和冲刺调整

## 常见错误

### 错误1：只看进度不看质量

**问题**：只评估任务完成率，不评估质量指标

**修正**：必须同时评估功能完整性和质量指标（错误、测试、性能）

### 错误2：决策模糊

**问题**：输出"基本可以"而非明确的"通过/条件通过/不通过"

**修正**：必须使用标准决策术语，明确说明条件或理由

### 错误3：忽略速度趋势

**问题**：只看当前冲刺速度，不看趋势

**修正**：分析速度趋势（改善/稳定/下降），基于趋势调整估算

### 错误4：范围建议无理由

**问题**：列出削减候选但不说明理由

**修正**：每个范围建议必须有明确理由（为什么保护/为什么削减）

### 错误5：忽略技术债务

**问题**：不扫描代码健康，忽略TODO/FIXME/HACK

**修正**：必须扫描代码健康，评估技术债务趋势
