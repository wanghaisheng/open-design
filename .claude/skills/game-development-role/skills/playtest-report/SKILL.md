# Playtest Report

生成结构化玩测报告模板或分析现有玩测笔记为结构化格式。用于标准化玩测反馈收集和分析。

## 适用场景

- 需要为玩测会话创建结构化报告模板
- 需要将原始玩测笔记分析整理为结构化格式
- 需要收集和标准化玩家反馈（定性+定量）
- 需要识别设计问题和验证设计假设
- 需要与设计师和制作人分享玩测发现

## 操作步骤

### 步骤1：创建玩测会话基础信息

**目标**：建立玩测会话的上下文和元数据

**操作方法**：
1. 记录日期和测试的游戏版本/提交哈希
2. 记录会话时长（实际游戏时间）
3. 记录测试者信息（匿名ID或姓名，经验水平）
4. 记录技术环境：
   - 平台（PC/主机/移动）
   - 输入方式（键鼠/手柄/触摸）
5. 确定会话类型：
   - 首次：新玩家第一次体验
   - 返回：老玩家再次体验
   - 目标测试：针对特定功能/流程的测试
6. 明确测试重点（本次会话要验证的具体功能或假设）

**检查清单**：
- [ ] 日期和版本已记录
- [ ] 测试者信息完整（匿名处理）
- [ ] 平台/输入方式已记录
- [ ] 会话类型已确定
- [ ] 测试重点明确

### 步骤2：收集第一印象（前5分钟）

**目标**：捕捉玩家首次接触游戏时的关键反应

**操作方法**：
1. 观察玩家是否理解游戏目标（通过提问或自我报告）
2. 观察玩家是否理解控制方式（记录困惑点）
3. 记录情感反应分类：
   - 投入：立即开始探索/尝试
   - 困惑：犹豫/反复尝试/表情困惑
   - 无聊：快速点击/跳过/目光游离
   - 沮丧：重复失败/负面语言
   - 兴奋：积极反应/赞叹
4. 记录关键观察（玩家说了什么/做了什么）
5. 在5分钟节点标记，切换到下阶段观察

**检查清单**：
- [ ] 目标理解情况已记录
- [ ] 控制理解情况已记录
- [ ] 情感反应已分类
- [ ] 关键言行已记录
- [ ] 5分钟节点已标记

### 步骤3：跟踪游戏流程体验

**目标**：系统性地记录玩家在整个游戏过程中的体验

**操作方法**：
1. 记录效果好的设计（玩家自然理解/享受的地方）
2. 记录痛点（玩家遇到困难或挫败的地方）：
   - 标记严重性：高（阻塞）/中（困扰）/低（轻微不便）
3. 记录困惑点（玩家不理解的地方和原因）
4. 记录惊喜时刻（让玩家愉悦或惊讶的设计）
5. 使用时间戳标记关键时刻（便于回顾录像）

**检查清单**：
- [ ] 有效设计已记录（带时间戳）
- [ ] 痛点已记录（带严重性）
- [ ] 困惑点已记录（带原因）
- [ ] 惊喜时刻已记录
- [ ] 时间戳准确

### 步骤4：记录遇到的Bug

**目标**：捕获技术问题和可重现性信息

**操作方法**：
1. 对每个Bug记录：
   - 描述（发生了什么）
   - 严重性（高/中/低）
   - 触发步骤（如何重现）
   - 可重现性（总是/经常/有时/一次）
2. 截图或录像作为证据
3. 标记Bug出现的游戏状态/位置
4. 询问玩家该Bug是否影响体验

**检查清单**：
- [ ] 每个Bug有描述和严重性
- [ ] 重现步骤已记录
- [ ] 可重现性已评估
- [ ] 证据（截图/录像）已收集

### 步骤5：收集特定功能反馈

**目标**：针对测试重点收集深入反馈

**操作方法**：
1. 针对测试重点中的每个功能：
   - 询问玩家是否理解功能目的（是/否/部分）
   - 询问玩家是否发现功能有趣（是/否）
   - 记录玩家的具体建议
2. 观察玩家使用功能时的行为（自然使用 vs 困惑）
3. 记录功能使用的频率和情境
4. 如果有替代方案，询问玩家偏好

**检查清单**：
- [ ] 每个测试功能都有反馈
- [ ] 理解度和趣味性已评估
- [ ] 具体建议已记录
- [ ] 使用行为已观察记录

### 步骤6：收集定量数据

**目标**：获取可统计的客观数据

**操作方法**：
1. 记录死亡次数和死亡位置（地图坐标/区域）
2. 记录每个区域的停留时间（识别卡点或快速通过区域）
3. 记录物品/技能使用情况（使用了什么/何时使用/未使用的）
4. 记录发现vs遗漏（玩家发现了哪些功能/错过了哪些）
5. 如果使用遥测数据，导出并关联到观察记录

**检查清单**：
- [ ] 死亡数据已记录
- [ ] 区域时间已记录
- [ ] 物品使用已记录
- [ ] 发现/遗漏已记录
- [ ] 遥测数据已关联（如有）

### 步骤7：获取整体评估

**目标**：收集玩家的总体感受和意愿

**操作方法**：
1. 询问玩家是否愿意再次玩（是/否/可能），并询问原因
2. 询问难度感受（太简单/正好/太难），并询问具体例子
3. 询问节奏感受（太慢/好/太快），并询问哪个部分
4. 询问会话长度偏好（更短/好/更长）
5. 开放性问题："如果只能改一件事，你会改什么？"

**检查清单**：
- [ ] 重玩意愿已记录（带原因）
- [ ] 难度评估已记录（带例子）
- [ ] 节奏评估已记录
- [ ] 长度偏好已记录
- [ ] 开放反馈已记录

### 步骤8：分析原始笔记并输出结构化报告

**目标**：将原始笔记整理为结构化格式，提取优先级发现

**操作方法**：
1. 如果是分析模式（已有笔记）：
   - 读取原始笔记
   - 提取关键发现，按报告模板分类
   - 交叉参考设计文档，标记冲突
2. 如果是新建模式（刚完成的测试）：
   - 直接使用收集的数据填充报告模板
3. 识别前3优先级发现：
   - 最严重的问题（阻塞/高严重性）
   - 最频繁的反馈（多个玩家提到）
   - 最容易修复的高影响问题
4. 确保每个发现都有证据支持（引用观察/录像/数据）
5. 输出结构化报告（使用报告模板格式）

**检查清单**：
- [ ] 原始数据已分类整理
- [ ] 前3优先级已确定
- [ ] 每个发现有证据支持
- [ ] 报告格式符合模板
- [ ] 已准备分享给团队

## 操作注意事项

### 观察者的中立性

1. **避免引导性提问**：不要问"你觉得这个很难吗？"，而是问"你在这个部分的感受如何？"
2. **记录事实而非解释**：记录"玩家尝试了3次后放弃"，而非"玩家不喜欢这个设计"
3. **区分观察与推断**：观察=玩家做了什么；推断=玩家为什么这么做（需要验证）

### 定性与定量的平衡

1. **定性数据**（感受/想法）提供深度，但主观性强
2. **定量数据**（死亡次数/时间）提供客观证据，但缺乏上下文
3. **最佳实践**：每个定量发现都配一个定性解释（例如："死亡3次（定量），因为玩家没有注意到陷阱（定性）"）

### 样本大小的局限性

1. **单个玩测会话的局限**：一个玩家的体验不能代表所有玩家
2. **趋势识别**：寻找多个会话中的共同模式
3. **极端案例**：注意异常值，但不要因为个别反馈改变设计

### 隐私和伦理

1. **匿名化处理**：报告中使用匿名ID，不记录真实姓名
2. **录像同意**：录像前获得玩家明确同意
3. **数据安全**：玩测数据存储在安全位置，限制访问权限

## 协作协议

### 向谁汇报

- **qa-lead**：报告的结构完整性和数据准确性
- **lead-designer**：设计问题和玩家反馈的解读
- **producer**：整体趋势和资源分配建议

### 协调对象

- **game-designer**：游戏玩法相关反馈的讨论和迭代
- **ux-designer**：用户体验问题的修复建议
- **ui-designer**：界面问题的修复建议
- **level-designer**：关卡设计问题的修复建议
- **qa-tester**：Bug验证和后续测试计划

## 常见错误

### 错误1：观察不够客观

**问题**：观察者记录"玩家不喜欢战斗系统"（主观推断），而非"玩家在第3次失败后退出战斗"（客观观察）

**修正**：记录可观察的行为和原话，将推断留给分析阶段

### 错误2：忽略沉默的玩家

**问题**：只记录主动说话的玩家，忽略沉默但表现困惑的玩家

**修正**：观察非语言信号（表情/操作犹豫/反复尝试），主动询问沉默玩家的感受

### 错误3：过度关注负面反馈

**问题**：只记录问题和痛点，忽略有效设计的正面证据

**修正**：同时记录"效果好的地方"和"惊喜时刻"，平衡报告

### 错误4：过早下结论

**问题**：基于一个玩家的反馈就得出结论（"所有玩家都会困惑"）

**修正**：使用"这个玩家..."而非"玩家..."，等待多个会话的趋势确认

### 错误5：混淆玩家问题与设计问题

**问题**：将玩家操作失败归因于"玩家不会玩"，而非设计问题

**修正**：假设如果玩家失败，责任在设计。记录"玩家未理解提示"而非"玩家笨"
