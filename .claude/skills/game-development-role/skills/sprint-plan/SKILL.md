# Sprint Plan

基于当前里程碑、已完成工作和可用容量生成新的冲刺计划或更新现有冲刺计划。从生产文档和设计待办事项中提取上下文。

## 适用场景

- 需要为新的冲刺生成计划
- 需要更新现有冲刺计划（添加/移除任务）
- 需要跟踪冲刺进度和生成状态报告
- 需要评估容量和选择任务
- 需要管理冲刺风险和依赖

## 操作步骤

### 步骤1：读取当前里程碑信息

**目标**：理解冲刺对里程碑的贡献

**操作方法**：
1. 读取 production/milestones/ 中的当前里程碑文档
2. 理解当前里程碑目标
3. 理解里程碑进度（已完成/剩余）
4. 识别里程碑的截止日期

**检查清单**：
- [ ] 当前里程碑已读取
- [ ] 里程碑目标已理解
- [ ] 里程碑进度已评估

### 步骤2：读取前一冲刺状态

**目标**：学习前一冲刺的经验和遗留

**操作方法**：
1. 读取 production/sprints/ 中的前一冲刺计划
2. 理解速度（计划 vs 实际）
3. 识别前一冲刺的遗留任务
4. 学习前一冲刺的经验（哪些假设正确/错误）

**检查清单**：
- [ ] 前一冲刺已读取
- [ ] 速度已计算
- [ ] 遗留任务已识别
- [ ] 经验已记录

### 步骤3：扫描设计文档识别就绪功能

**目标**：识别准备好实现的功能

**操作方法**：
1. 扫描 design/gdd/ 中的设计文档
2. 识别准备好实现的功能（GDD已通过设计审查）
3. 评估设计完成度（8个必需部分存在）
4. 按优先级排序功能（基于里程碑贡献）

**检查清单**：
- [ ] 设计文档已扫描
- [ ] 就绪功能已识别
- [ ] 设计完成度已评估
- [ ] 优先级已排序

### 步骤4：检查风险登记册

**目标**：识别相关风险并评估影响

**操作方法**：
1. 读取 production/risk-register/
2. 识别与当前冲刺相关的风险
3. 评估风险概率和影响
4. 制定缓解措施

**检查清单**：
- [ ] 风险登记册已读取
- [ ] 相关风险已识别
- [ ] 缓解措施已制定

### 步骤5：计算可用容量

**目标**：确定冲刺的实际可用容量

**操作方法**：
1. 确定冲刺时长（通常2周）
2. 计算总天数（工作日）
3. 预留缓冲（20%为未计划工作）
4. 计算可用容量（总天数 - 缓冲）
5. 考虑团队可用性（假期、会议）

**检查清单**：
- [ ] 冲刺时长已确定
- [ ] 总天数已计算
- [ ] 缓冲已预留（20%）
- [ ] 可用容量已计算
- [ ] 团队可用性已考虑

### 步骤6：选择任务并分类

**目标**：选择适合容量的任务

**操作方法**：
1. **必须有（关键路径）**：必须完成的任务，影响里程碑目标
2. **应该有**：重要但非关键的任务
3. **可以有**：锦上添花的任务
4. 使用 /estimate 为每个任务生成估算
5. 确保任务总估算在可用容量内
6. 如果超出容量，推迟低优先级任务

**检查清单**：
- [ ] 任务已分类（必须有/应该有/可以有）
- [ ] 每个任务已估算
- [ ] 总估算在容量内
- [ ] 低优先级任务已推迟（如需要）

### 步骤7：识别和管理依赖

**目标**：确保任务依赖关系清晰

**操作方法**：
1. 识别前一冲刺的遗留任务
2. 识别外部依赖（外部因素、第三方）
3. 识别任务之间的依赖关系
4. 在任务表中记录依赖
5. 识别关键路径（依赖链最长的任务）

**检查清单**：
- [ ] 遗留任务已识别
- [ ] 外部依赖已识别
- [ ] 任务依赖已记录
- [ ] 关键路径已识别

### 步骤8：定义冲刺目标

**目标**：用一句话描述冲刺目标

**操作方法**：
1. 描述此冲刺对里程碑的贡献
2. 确保目标简洁（一句话）
3. 确保目标可衡量（有明确的成功标准）
4. 与团队确认目标

**检查清单**：
- [ ] 目标已定义
- [ ] 目标简洁（一句话）
- [ ] 目标可衡量

### 步骤9：生成冲刺计划文档

**目标**：输出结构化冲刺计划

**操作方法**：
1. 输出冲刺目标
2. 输出容量信息（总天数、缓冲、可用）
3. 输出任务表（必须有/应该有/可以有）
4. 输出前一冲刺遗留
5. 输出风险表
6. 输出外部因素依赖
7. 输出完成定义

**检查清单**：
- [ ] 冲刺目标已输出
- [ ] 容量信息已输出
- [ ] 任务表已输出
- [ ] 风险已输出
- [ ] 完成定义已输出

### 步骤10：生成状态报告（更新现有冲刺）

**目标**：跟踪冲刺进度

**操作方法**：
1. 跟踪进度（X/Y任务完成）
2. 列出已完成任务
3. 列出进行中任务（%完成、阻塞）
4. 列出未开始任务
5. 列出已阻塞任务
6. 评估燃尽（按计划/落后/超前）
7. 识别新出现的风险

**检查清单**：
- [ ] 进度已跟踪
- [ ] 任务状态已更新
- [ ] 燃尽已评估
- [ ] 新风险已识别

## 操作注意事项

### 容量规划的现实性

1. **包含缓冲**：始终预留20%缓冲为未计划工作
2. **考虑团队可用性**：考虑假期、会议等非工作时间
3. **避免过度承诺**：确保任务总估算在可用容量内

### 任务优先级管理

1. **必须有优先**：必须有任务影响里程碑目标，必须完成
2. **应该有其次**：重要但非关键的任务
3. **可以有最后**：锦上添花的任务，时间允许才做

### 依赖管理的重要性

1. **识别关键路径**：依赖链最长的任务决定冲刺完成时间
2. **记录外部依赖**：外部因素、第三方依赖需要特别关注
3. **处理遗留任务**：前一冲刺的遗留任务优先处理

### 冲刺目标的明确性

1. **一句话描述**：目标应简洁，一句话可说完
2. **可衡量**：目标应有明确的成功标准
3. **里程碑贡献**：目标应描述对里程碑的贡献

## 协作协议

### 向谁汇报

- **producer**：冲刺计划和容量规划
- **scrum-master**：冲刺状态和燃尽评估
- **technical-lead**：技术任务估算和依赖

### 协调对象

- **game-designer**：功能优先级和设计准备度评估
- **gameplay-programmer**：任务估算和可行性验证
- **qa-lead**：测试任务估算和验收标准确认

## 常见错误

### 错误1：不预留缓冲时间

**问题**：将所有可用时间分配给任务，不预留缓冲

**修正**：始终预留20%缓冲为未计划工作，计算可用容量 = 总天数 - 缓冲

### 错误2：任务估算不准确

**问题**：凭空估算任务，不使用 /estimate 技能

**修正**：使用 /estimate 为每个任务生成带范围的结构化估算

### 错误3：忽略依赖关系

**问题**：不记录任务依赖，导致关键路径阻塞

**修正**：识别并记录所有依赖关系，识别关键路径

### 错误4：冲刺目标模糊

**问题**：目标如"做一些工作"或"实现功能"

**修正**：目标应简洁（一句话）、可衡量（有成功标准）、描述里程碑贡献

### 错误5：燃尽评估不准确

**问题**：只看任务完成数量，不考虑任务大小

**修正**：燃尽评估应基于估算工作量，而非任务数量
