# Concept Art

创建游戏概念艺术，包括角色概念、环境概念、物品概念和UI概念，为游戏提供视觉方向和风格参考。

## 适用场景

- 需要创建角色概念设计
- 需要创建环境概念设计
- 需要创建物品/武器/装备概念
- 需要创建UI概念设计
- 需要建立游戏的视觉方向和风格参考

## 操作步骤

### 步骤1：需求分析

**目标**：理解游戏需求和设计意图

**操作方法**：
1. 读取相关设计文档（GDD、角色设计、关卡设计）
2. 理解角色/环境/物品的功能和定位
3. 确定视觉风格（写实、卡通、像素等）
4. 识别技术限制（多边形数量、纹理分辨率）

**检查清单**：
- [ ] 设计文档已读取
- [ ] 功能和定位已理解
- [ ] 视觉风格已确定
- [ ] 技术限制已识别

### 步骤2：资料收集和情绪板建立

**目标**：收集参考并建立视觉方向

**操作方法**：
1. 收集参考图（类似游戏、艺术作品、照片）
2. 收集灵感（色彩、构图、风格）
3. 建立情绪板（使用 PureRef 或类似工具）
4. 按主题分类参考（角色、环境、道具等）

**检查清单**：
- [ ] 参考图已收集
- [ ] 灵感已收集
- [ ] 情绪板已建立
- [ ] 参考已分类

### 步骤3：草图绘制和方案探索

**目标**：探索不同设计方向

**操作方法**：
1. 绘制多个草图（至少3个不同方案）
2. 探索不同构图和姿态
3. 探索不同风格元素
4. 与设计团队讨论，确定方向

**检查清单**：
- [ ] 多个草图已绘制
- [ ] 不同方案已探索
- [ ] 方向已确定

### 步骤4：细化设计

**目标**：将选定方案细化为完整设计

**操作方法**：
1. 细化选定草图
2. 添加细节（纹理、光影、材质）
3. 完善设计（比例、解剖、透视）
4. 确保视觉一致性（与游戏整体风格一致）

**检查清单**：
- [ ] 草图已细化
- [ ] 细节已添加
- [ ] 设计已完善
- [ ] 视觉一致性已检查

### 步骤5：最终渲染

**目标**：生成高质量的概念图

**操作方法**：
1. 最终渲染（使用 Photoshop/Procreate/Krita）
2. 调整色彩（确保色彩一致）
3. 添加特效（光影、氛围）
4. 确保可转化为3D（考虑建模需求）

**检查清单**：
- [ ] 最终渲染已完成
- [ ] 色彩已调整
- [ ] 特效已添加
- [ ] 3D转化可行性已考虑

### 步骤6：编写设计文档

**目标**：记录设计决策和技术规格

**操作方法**：
1. 编写设计说明（设计意图、风格选择）
2. 记录设计决策（为什么选择这个方案）
3. 提供技术规格（尺寸、材质、色彩规范）
4. 提供参考图集（用于后续制作）

**检查清单**：
- [ ] 设计说明已编写
- [ ] 设计决策已记录
- [ ] 技术规格已提供
- [ ] 参考图集已整理

## 操作注意事项

### 视觉一致性

1. **风格一致**：确保所有概念图与游戏整体风格一致
2. **色彩一致**：使用统一的色彩规范和调色板
3. **比例一致**：确保角色/环境/物品的比例关系一致

### 功能性考虑

1. **符合游戏需求**：概念图必须服务于游戏玩法需求
2. **符合技术限制**：考虑多边形数量、纹理分辨率等技术限制
3. **符合性能要求**：避免过于复杂的设计影响性能

### 可执行性

1. **可转化为3D**：概念图应考虑建模需求，避免无法实现的细节
2. **可实现**：设计应在团队技术能力范围内
3. **可优化**：设计应考虑后续优化空间

### 创意与实用的平衡

1. **独特创意**：追求独特和创新的设计
2. **视觉冲击**：确保设计有视觉吸引力
3. **实用优先**：创意不能牺牲可实现性

## 协作协议

### 向谁汇报

- **art-director**：概念图和视觉方向
- **lead-designer**：设计与视觉的一致性
- **technical-director**：技术规格和可行性

### 协调对象

- **character-artist**：角色概念细节和建模需求
- **environment-artist**：环境概念细节和布局
- **ui-designer**：UI概念和交互设计

## 常见错误

### 错误1：忽略技术限制

**问题**：设计过于复杂，超出技术限制（多边形、纹理）

**修正**：在设计前明确技术限制，确保设计在限制内

### 错误2：视觉不一致

**问题**：不同概念图风格不一致（如写实与卡通混用）

**修正**：建立统一的视觉风格指南，确保所有概念图一致

### 错误3：忽略游戏需求

**问题**：概念图很漂亮但不服务于游戏玩法

**修正**：始终从游戏需求出发，概念图必须支持玩法

### 错误4：无法转化为3D

**问题**：概念图细节过于复杂，无法建模

**修正**：在设计时考虑建模需求，避免无法实现的细节

### 错误5：缺少设计文档

**问题**：只有概念图，没有设计说明和技术规格

**修正**：必须编写设计文档，记录设计决策和技术规格
