# Core Loop Design

设计游戏核心循环，定义玩家在游戏中的主要行为模式和奖励机制。

## 适用场景

- 从 0 到 1 设计游戏核心循环（动作-奖励-进展-挑战）
- 需要把核心循环拆解为可执行的行为步骤、奖励时机与进展曲线
- 核心循环出现玩家不知道做什么、奖励不合理、循环枯燥等问题，需要迭代优化
- 需要与系统设计/数值平衡/关卡设计协作，明确循环与系统/关卡的结合点

## 操作步骤

### 步骤1：把核心循环目标写成“可验收”的一句话

**目标**：明确核心循环服务什么体验目标、成功标准是什么

**操作方法**：
1. 写核心循环主题与一句话目标（面向玩家动机）
2. 列出循环输入/输出（玩家行为、资源变化、状态变化）
3. 写 3-5 条成功标准（例如：玩家能在 30 秒内理解循环；单次循环 3-5 分钟有满足感）
4. 写 3-5 条失败信号（例如：玩家不知道下一步做什么；循环重复后失去兴趣）

**检查清单**：
- [ ] 有清晰边界：循环“包含什么/不包含什么”
- [ ] 有成功标准：可观测、可测试
- [ ] 有失败信号：能快速触发调整

### 步骤2：用“行为-奖励-进展-挑战”四块拆解循环

**目标**：把循环拆成可实现的设计单元，避免只写概念

**操作方法**：
1. **行为（Action）**：列核心行为（每条写输入方式、反馈机制、技能要求）
2. **奖励（Reward）**：列即时奖励/延迟奖励（每条写触发时机、奖励类型、数值区间）
3. **进展（Progression）**：列短期/中期/长期进展（每条写进展指标、解锁条件、视觉反馈）
4. **挑战（Challenge）**：列技能/策略/运气挑战（每条写难度区间、失败惩罚、重试成本）

**检查清单**：
- [ ] 每个行为都有输入方式与反馈机制
- [ ] 每个奖励都有触发时机与数值区间
- [ ] 每个进展都有解锁条件与视觉反馈
- [ ] 每个挑战都有难度区间与失败惩罚

### 步骤3：画循环流程图并标注节奏

**目标**：把循环可视化，确保节奏合理

**操作方法**：
1. 画循环流程图（行为 → 奖励 → 进展 → 挑战 → 行为）
2. 标注每个环节的时长（例如：行为 30 秒、奖励 5 秒、进展 10 秒）
3. 标注关键决策点（玩家需要做出选择的地方）
4. 标注循环变体（不同情况下的循环分支）

**检查清单**：
- [ ] 循环流程图完整且闭环
- [ ] 每个环节时长合理
- [ ] 关键决策点明确
- [ ] 循环变体可维护

### 步骤4：定义循环与系统/关卡的结合点

**目标**：让循环与系统/关卡结合，不割裂体验

**操作方法**：
1. 列循环与系统的结合点（经济系统、成长系统、社交系统）
2. 列循环与关卡的结合点（任务、遭遇、探索）
3. 写循环在不同阶段的强度曲线（新手期/成长期/成熟期）
4. 写循环的退出/重入机制（避免玩家卡在循环中）

**检查清单**：
- [ ] 循环与系统结合点明确
- [ ] 循环与关卡结合点明确
- [ ] 强度曲线合理
- [ ] 有退出/重入机制

### 步骤5：输出循环规格并组织测试迭代

**目标**：让程序与设计能直接落地，并能高效迭代

**操作方法**：
1. 输出循环流程图（行为-奖励-进展-挑战）
2. 输出参数表（时长、数值区间、解锁条件）
3. 输出测试用例（新手理解度、循环满足感、长期可玩性）
4. 收集数据与反馈（循环次数、停留时长、流失点）并形成迭代记录

**检查清单**：
- [ ] 循环流程图完整
- [ ] 参数表可配置且有默认值/范围
- [ ] 测试用例可复现
- [ ] 有迭代记录（问题-假设-改动-结果）

## 操作注意事项

### 清晰性与引导

1. 玩家必须在 30 秒内理解循环：行为-奖励-进展-挑战。
2. 关键决策点必须有明确提示，不要让玩家困惑。

### 满足感与平衡

1. 即时奖励必须有视觉/听觉反馈，让玩家“感觉到”奖励。
2. 奖励与付出必须匹配，避免“努力很多、奖励很少”或“努力很少、奖励很多”。

### 可重复性与变化

1. 循环必须有变化（随机性、分支、解锁），避免纯重复导致枯燥。
2. 长期可玩性必须依赖进展系统，而不是单纯重复。

## 输出格式

### 核心循环文档

- 核心循环目标、约束、验收标准、失败信号
- 循环流程图（行为-奖励-进展-挑战）
- 行为表（输入方式、反馈机制、技能要求）
- 奖励表（触发时机、奖励类型、数值区间）
- 进展表（短期/中期/长期进展、解锁条件、视觉反馈）
- 挑战表（技能/策略/运气挑战、难度区间、失败惩罚）
- 循环与系统/关卡结合点
- 强度曲线（新手期/成长期/成熟期）
- 测试用例与迭代记录

## 协作协议

### 向谁汇报

- lead-designer：循环与整体体验目标对齐
- technical-director：循环可实现性与性能约束

### 协调对象

- system-designer：循环与系统结合点（经济/成长/社交）
- level-designer：循环与关卡结合点（任务/遭遇/探索）
- balance-designer：数值平衡与难度曲线
- qa-tester：循环测试、数据收集、迭代验证

## 常见错误

### 错误1：只写“主题/氛围”，没有具体的循环流程与参数

**问题**：实现阶段无法拆分任务，循环容易空洞或不可落地。

**修正**：按步骤2/3补齐循环流程图与参数表。

### 错误2：奖励与付出不匹配，玩家感觉“不值得”

**问题**：玩家失去动力，不再重复循环。

**修正**：在步骤2重新评估奖励数值区间，确保与付出匹配。

### 错误3：循环没有变化，长期重复导致枯燥

**问题**：玩家在重复多次后失去兴趣，留存下降。

**修正**：在步骤3/4增加循环变体（随机性、分支、解锁）与进展系统。
