# Design Systems

将游戏概念分解为独立系统，映射依赖关系，优先确定设计顺序，指导创建每个系统的GDD。

## 系统分解要素

### 1. 系统枚举 (Systems Enumeration)
- 显式系统：从概念文档直接提到的系统
- 隐式系统：每个显式系统隐含的隐藏系统
- 系统分类：按类别组织系统
- 系统描述：每个系统的简要描述

### 2. 依赖映射 (Dependency Mapping)
- 输入/输出依赖：系统A产生系统B需要的数据
- 结构依赖：系统A提供系统B插入的框架
- UI依赖：每个游戏玩法系统都有对应的UI系统
- 依赖排序：将系统排列到层级中

### 3. 优先级分配 (Priority Assignment)
- MVP：概念中MVP部分提到的系统
- 垂直切片：在一个区域提供完整体验所需的系统
- Alpha：所有剩余的游戏玩法系统
- 完整愿景：润色、元系统和nice-to-have系统

### 4. 设计顺序 (Design Order)
- MVP基础系统：首先设计和构建
- MVP核心系统：第二
- MVP功能系统：第三
- 垂直切片基础/核心系统
- 依此类推

## 依赖层级

### 1. 基础层 (Foundation)
- 零依赖的系统
- 首先设计和构建
- 提供框架基础

### 2. 核心层 (Core)
- 仅依赖基础层的系统
- 第二优先级
- 核心游戏玩法

### 3. 功能层 (Feature)
- 依赖核心层的系统
- 第三优先级
- 具体游戏功能

### 4. 表现层 (Presentation)
- 包装游戏玩法的UI和反馈系统
- 在游戏玩法系统之后设计
- 视觉和音频反馈

### 5. 润色层 (Polish)
- 元系统、教程、分析、可访问性
- 最后设计
- 体验增强

## 隐式系统推理

### 常见隐含系统
- **物品栏**：物品数据库、装备槽、重量/容量规则、物品UI、保存/加载的物品序列化
- **战斗**：伤害计算、生命系统、命中检测、状态效果、敌人AI、战斗UI、死亡/重生
- **开放世界**：流式/分块、LOD系统、快速旅行、地图/小地图、兴趣点跟踪、世界状态持久化
- **多人游戏**：网络层、大厅/匹配、状态同步、反作弊、网络UI
- **制作**：配方数据库、原料收集、制作UI、成功/失败机制、配方发现/学习
- **对话**：对话树系统、对话UI、选择跟踪、NPC状态管理、本地化挂钩
- **进度**：XP系统、升级机制、技能树、解锁跟踪、进度UI、进度保存数据

## 分解原则

### 1. 协作设计
- 用户驱动协作
- 在每个决策点提问
- 逐步推进决策

### 2. 依赖管理
- 识别依赖关系
- 检测循环依赖
- 解决瓶颈系统

### 3. 优先级管理
- 基于MVP分配优先级
- 考虑依赖顺序
- 平衡风险和收益

### 4. 质量保证
- 设计审查循环
- 每个GDD通过设计审查
- 早期发现问题

## 分解流程

### 阶段1：读取概念（必需上下文）
- 读取游戏概念
- 读取游戏支柱（如果存在）
- 读取系统索引（如果存在）
- 检查现有GDD

### 阶段2：系统枚举（协作）
- 提取显式系统
- 识别隐式系统
- 用户审查
- 迭代直到批准

### 阶段3：依赖映射（协作）
- 映射依赖
- 按依赖排序
- 检测循环依赖
- 呈现给用户审查

### 阶段4：优先级分配（协作）
- 基于概念自动分配
- 用户审查
- 确定设计顺序
- 解释推理

### 阶段5：创建系统索引（写入）
- 起草文档
- 用户批准
- 更新会话状态
- 写入文件

### 阶段6：设计单个系统（可选，迭代）
- 选择系统
- 委托给代理
- 编写GDD
- 运行设计审查
- 更新系统索引
- 循环或停止

### 阶段7：建议后续步骤
- 推荐适当的代理
- 推荐设计审查
- 推荐原型
- 推荐冲刺规划
