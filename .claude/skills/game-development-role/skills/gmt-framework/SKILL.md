# GMT Framework

GMT框架应用技能，运用目标-手段-工具框架拆解玩家目标和体验目标。

## 适用场景

- 拆解游戏设计目标
- 构建游戏设计路径
- 分析游戏设计逻辑
- 优化游戏设计
- 确保设计一致性

## 操作步骤

### 步骤1：区分三种目标

**目标**：清晰区分玩家目标、体验目标和设计目标

**操作方法**：
1. **玩家目标（Player Goal）**
   - 定义：游戏要求玩家达成的结果
   - 特点：最表层、最机制化
   - 示例："击败最终Boss"、"在规定时间内到达终点"

2. **体验目标（Experience Goal）**
   - 定义：设计师期望玩家获得的反馈
   - 特点：设计的核心、"灵魂"
   - 示例："让玩家体验到成长的喜悦"、"营造紧张感"

3. **设计目标（Design Goal）**
   - 定义：设计师希望达成的特定效果
   - 特点：关乎系统、产品或个人表达
   - 示例："设计易于上手难于精通的战斗系统"

**检查清单**：
- [ ] 玩家目标已定义
- [ ] 体验目标已定义
- [ ] 设计目标已定义
- [ ] 目标区分已明确

### 步骤2：应用玩家GMT

**目标**：用GMT框架拆解玩家目标

**操作方法**：
1. **G（Goal - 目标）**
   - 定义最终要达成的目的
   - 示例："到达关卡终点"

2. **M（Means - 手段）**
   - 定义高阶策略或维度
   - 示例："平台移动、战胜敌人"

3. **T（Tools - 工具）**
   - 定义低阶可执行元素
   - 示例："跑、跳、踩、顶砖块、吃蘑菇"

4. **推导方式**
   - 自上而下（G→M→T）：先设定目标，再思考手段，最后设计工具
   - 自下而上（T/M→G）：从有趣工具出发，反推出目标

**检查清单**：
- [ ] 玩家目标已定义
- [ ] 手段已定义
- [ ] 工具已定义
- [ ] 推导方式已选择

### 步骤3：应用体验GMT

**目标**：用GMT框架拆解体验目标

**操作方法**：
1. **G（Goal - 目标）**
   - 定义最终要达成的体验
   - 示例："让玩家体验末世中的人性、情感羁绊和残酷"

2. **M（Means - 手段）**
   - 定义高阶实现途径
   - 示例："电影化的叙事、资源匮乏带来的生存压力、角色塑造"

3. **T（Tools - 工具）**
   - 定义低阶可感知内容
   - 示例："精心编写的对白、过场动画、稀缺的弹药和物资"

4. **推导方式**
   - 自上而下（G→M→T）：先设定体验，再思考手段，最后设计工具
   - 自下而上（T/M→G）：从有趣工具出发，反推出体验

**检查清单**：
- [ ] 体验目标已定义
- [ ] 手段已定义
- [ ] 工具已定义
- [ ] 推导方式已选择

### 步骤4：M-T拆分应用

**目标**：将手段和工具分开，控制设计范围

**操作方法**：
1. **M-T分开的重要性**
   - 先就高阶方向（M）达成共识
   - 避免过早陷入细节（T）争论
   - 控制设计范围和试错成本

2. **团队共识建立**
   - 在项目早期就高阶方向达成共识
   - 例如："我们要做强调'智力考验'的游戏"
   - 避免陷入"具体要做哪种谜题"的细节

3. **设计范围控制**
   - 明确手段（M）范围
   - 控制工具（T）数量
   - 避免设计过度复杂

**检查清单**：
- [ ] M-T分开已理解
- [ ] 团队共识已建立
- [ ] 设计范围已控制
- [ ] 试错成本已降低

### 步骤5：验证GMT应用

**目标**：验证GMT框架应用的有效性

**操作方法**：
1. **一致性验证**
   - 验证工具是否服务于手段
   - 验证手段是否服务于目标
   - 验证整体逻辑一致性

2. **完整性验证**
   - 验证目标是否明确
   - 验证手段是否充分
   - 验证工具是否具体

3. **可执行性验证**
   - 验证工具是否可执行
   - 验证手段是否可实现
   - 验证目标是否可达成

**检查清单**：
- [ ] 一致性已验证
- [ ] 完整性已验证
- [ ] 可执行性已验证
- [ ] GMT应用已验证

## 操作注意事项

### 目标区分

1. **清晰区分**：三种目标要清晰区分
2. **无关性**：三种目标之间没有必然逻辑捆绑
3. **创作自由**：无关性赋予创作自由
4. **体验优先**：体验目标应是首要出发点

### GMT应用

1. **层级清晰**：G、M、T层级要清晰
2. **自上而下**：通常采用自上而下推导
3. **M-T分开**：手段和工具要分开
4. **可追溯**：每个元素都能追溯到目标

### 团队协作

1. **早期共识**：早期就高阶方向达成共识
2. **避免细节**：避免过早陷入细节争论
3. **控制范围**：控制设计范围
4. **降低成本**：降低试错成本

## 常见错误

### 错误1：目标混淆

**问题**：三种目标混淆，设计混乱

**修正**：清晰区分三种目标，明确各自作用

### 错误2：M-T不分开

**问题**：手段和工具不分开，过早陷入细节

**修正**：将手段和工具分开，先达成手段共识

### 错误3：层级混乱

**问题**：G、M、T层级混乱，逻辑不清

**修正**：明确G、M、T层级，确保逻辑清晰

### 错误4：不验证一致性

**问题**：不验证工具是否服务于目标

**修正**：验证工具→手段→目标的一致性

### 错误5：不控制范围

**问题**：不控制设计范围，设计过度复杂

**修正**：通过M-T分开控制设计范围
