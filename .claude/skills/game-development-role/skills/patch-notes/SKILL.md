# Patch Notes

从git历史、冲刺数据和内部变更日志生成面向玩家的补丁说明。将开发者语言翻译为清晰、引人入胜的玩家沟通。

## 适用场景

- 需要生成补丁说明
- 需要翻译开发者语言为玩家语言
- 需要分类变更
- 需要生成不同风格的补丁说明
- 需要保存补丁说明

## 操作步骤

### 步骤1：解析参数

**目标**：解析参数

**操作方法**：
1. 版本：生成说明的发布版本
2. 风格：输出风格——简洁、详细、完整

**检查清单**：
- [ ] 版本已解析
- [ ] 风格已解析

### 步骤2：收集变更数据

**目标**：从多个来源收集变更数据

**操作方法**：
1. 读取内部变更日志（Changelog）
2. 运行git日志（Git）
3. 读取冲刺回顾（Sprint Retrospectives）
4. 读取平衡变更文档（Balance Documents）
5. 读取QA错误修复记录

**检查清单**：
- [ ] 内部变更日志已读取
- [ ] git日志已运行
- [ ] 冲刺回顾已读取
- [ ] 平衡变更文档已读取
- [ ] QA错误修复记录已读取

### 步骤3：分类变更

**目标**：分类所有变更

**操作方法**：
1. 新内容（新功能、地图、角色、物品、模式）
2. 游戏玩法变更（平衡调整、机制变更、进度变更）
3. 生活质量（UI改进、便利功能、可访问性）
4. 错误修复（按系统分组：战斗、UI、网络等）
5. 性能（玩家可能注意到的优化改进）
6. 已知问题（关于未解决问题的透明度）

**检查清单**：
- [ ] 新内容已分类
- [ ] 游戏玩法变更已分类
- [ ] 生活质量已分类
- [ ] 错误修复已分类
- [ ] 性能已分类
- [ ] 已知问题已分类

### 步骤4：翻译语言

**目标**：翻译开发者语言为玩家语言

**操作方法**：
1. 技术术语转换为玩家友好语言（"重构伤害计算管道" → "改进命中检测准确性"）
2. 移除不影响玩家的纯内部变更
3. 保留平衡变更的具体数字（伤害：50 → 45）
4. 描述玩家体验而非技术原因

**检查清单**：
- [ ] 技术术语已转换
- [ ] 纯内部变更已移除
- [ ] 平衡变更具体数字已保留
- [ ] 玩家体验已描述

### 步骤5：生成补丁说明

**目标**：使用适当风格生成补丁说明

**操作方法**：
1. 简洁风格（要点）
2. 详细风格（带上下文）
3. 完整风格（带开发者评论）

**检查清单**：
- [ ] 简洁风格已生成
- [ ] 详细风格已生成
- [ ] 完整风格已生成

### 步骤6：审查输出

**目标**：审查输出

**操作方法**：
1. 无内部术语
2. 无内部系统引用
3. 平衡变更包含前后值
4. 错误修复描述玩家体验
5. 语气匹配游戏声音

**检查清单**：
- [ ] 无内部术语
- [ ] 无内部系统引用
- [ ] 平衡变更包含前后值
- [ ] 错误修复描述玩家体验
- [ ] 语气匹配游戏声音

### 步骤7：保存补丁说明

**目标**：保存补丁说明

**操作方法**：
1. 保存到 production/releases/[version]/patch-notes.md
2. 创建目录（如需要）

**检查清单**：
- [ ] 补丁说明已保存
- [ ] 目录已创建

## 输出格式

### 简洁风格
```markdown
# 补丁 [版本] — [标题]

**新增**
- [功能1]
- [功能2]

**变更**
- [平衡/机制变更，带前后值]

**修复**
- [错误修复1]
- [错误修复2]

**已知问题**
- [问题1]
```

### 详细风格
```markdown
# 补丁 [版本] — [标题]
*[日期]*

## 亮点
[1-2句最激动人心的变更摘要]

## 新内容
### [功能名称]
[2-3句描述功能和玩家应兴奋的原因]

## 游戏玩法变更
### 平衡
| 变更 | 之前 | 之后 | 原因 |
| ---- | ---- | ---- | ---- |
| [物品/能力] | [旧值] | [新值] | [简要理由] |

### 机制
- **[变更]**：[变更解释和原因]

## 生活质量
- [带上下文的改进]

## 错误修复
### 战斗
- 修复[玩家经历的描述]

### UI
- 修复[描述]

### 网络
- 修复[描述]

## 性能
- [玩家将注意到的改进]

## 已知问题
- [问题和可用变通方法]
```

### 完整风格
包含详细风格的所有内容，加上：
```markdown
## 开发者评论
### [主题]
> [对重大变更的开发者洞察——为什么做出、考虑了什么、团队学到了什么。以第一人称团队声音编写。]
```

## 规则

- 面向玩家补丁说明中无内部术语（用玩家友好语言替换技术术语）
- 无内部系统、票据或冲刺编号引用
- 平衡变更包含前后值
- 错误修复描述玩家体验，而非技术原因
- 语气匹配游戏声音（根据游戏风格调整正式程度）
- 保存到 production/releases/[version]/patch-notes.md

## 操作注意事项

### 玩家友好

1. **避免内部术语**：避免内部术语
2. **专注于玩家体验**：专注于玩家体验
3. **使用清晰语言**：使用清晰语言

### 透明度

1. **诚实面对已知问题**：诚实面对已知问题
2. **提供变通方法**：提供变通方法
3. **解释变更原因**：解释变更原因

### 准确性

1. **基于实际变更**：基于实际变更
2. **验证变更**：验证变更

### 语气匹配

1. **匹配游戏声音**：匹配游戏声音

## 协作协议

### 向谁汇报

- **producer**：补丁说明内容和发布
- **community-manager**：玩家沟通
- **qa-lead**：错误修复验证

### 协调对象

- **developer**：变更详情
- **balance-designer**：平衡变更
- **writer**：语言翻译

## 常见错误

### 错误1：不收集变更数据

**问题**：不读取内部变更日志、git日志、冲刺回顾

**修正**：必须读取内部变更日志、git日志、冲刺回顾、平衡变更文档、QA错误修复记录

### 错误2：不分类变更

**问题**：不分类变更，直接生成补丁说明

**修正**：必须分类变更（新内容、游戏玩法变更、生活质量、错误修复、性能、已知问题）

### 错误3：不翻译语言

**问题**：不翻译开发者语言为玩家语言

**修正**：必须翻译开发者语言为玩家语言，技术术语转换为玩家友好语言

### 错误4：不审查输出

**问题**：不审查输出，包含内部术语

**修正**：必须审查输出，确保无内部术语、无内部系统引用

### 错误5：不保存补丁说明

**问题**：不保存补丁说明到指定路径

**修正**：必须保存补丁说明到 production/releases/[version]/patch-notes.md
