# Estimate

通过分析复杂性、依赖关系、历史速度和风险因素来估算任务工作量。生成带置信度水平的结构化估算。

## 估算要素

### 1. 代码复杂性 (Code Complexity)
- 受影响文件的代码行数
- 依赖数量和耦合级别
- 是否涉及核心/引擎代码 vs 叶子/功能代码
- 是否可以遵循现有模式或需要新模式

### 2. 范围 (Scope)
- 受影响的系统数量
- 新代码 vs 修改现有代码
- 需要的新测试覆盖量
- 需要的数据迁移或配置更改

### 3. 风险 (Risk)
- 新技术或不熟悉的库
- 不清楚或模糊的需求
- 对未完成工作的依赖
- 跨系统集成复杂性
- 性能敏感性

### 4. 历史数据 (Historical Data)
- 类似已完成任务的实际工作量
- 历史速度（计划 vs 实际）
- 估算偏差模式（持续高估或低估）

## 估算原则

### 1. 范围估算
- 始终给出范围（乐观/预期/悲观）
- 从不给出单一数字
- 单点估算产生虚假精度

### 2. 现实性
- 推荐预算应为预期估算
- 不应是最乐观的
- 填充不是不诚实，而是现实

### 3. 明确性
- 明确包含和排除的内容
- 范围模糊是估算错误的最常见来源
- 四舍五入到半天增量

### 4. 风险管理
- 如果置信度低，建议先进行时间框定的探索
- 不要默默填充估算
- 明确指出风险因素

## 估算流程

### 1. 读取任务描述
- 读取任务描述
- 如果描述太模糊无法有意义估算，要求澄清

### 2. 读取项目上下文
- 读取 CLAUDE.md
- 理解技术栈
- 理解编码标准
- 理解架构模式

### 3. 读取相关设计文档
- 读取 design/gdd/ 中的设计文档
- 理解相关功能或系统

### 4. 扫描代码库
- 识别需要更改的文件和模块
- 评估这些文件的复杂性
- 识别与其他系统的集成点
- 检查受影响区域的现有测试覆盖

### 5. 读取历史冲刺数据
- 查找类似已完成任务
- 计算历史速度
- 识别估算偏差模式

### 6. 分析因素
- 分析代码复杂性
- 分析范围
- 分析风险

### 7. 生成估算
- 生成结构化估算
- 包含置信度水平
- 包含风险因素
- 包含依赖关系

### 8. 输出估算摘要
- 输出推荐预算
- 输出置信度水平
- 输出最大风险因素
