# Estimate

通过分析复杂性、依赖关系、历史速度和风险因素来估算任务工作量。生成带置信度水平的结构化估算。

## 适用场景

- 需要为新任务生成工作量估算
- 需要评估任务的复杂性和风险
- 需要基于历史数据提高估算准确性
- 需要为冲刺规划提供输入
- 需要识别任务依赖关系和阻塞

## 操作步骤

### 步骤1：读取并理解任务描述

**目标**：建立对任务的清晰理解

**操作方法**：
1. 读取任务描述
2. 如果描述太模糊无法有意义估算，要求澄清
3. 用1-2句话重述任务以确认理解
4. 识别任务类型（新功能/修复/重构/测试）

**检查清单**：
- [ ] 任务描述已理解
- [ ] 任务类型已识别
- [ ] 如有模糊，已要求澄清

### 步骤2：读取项目上下文

**目标**：理解技术约束和编码标准

**操作方法**：
1. 读取 CLAUDE.md 了解技术栈
2. 读取 .claude/docs/technical-preferences.md 了解编码标准
3. 理解架构模式和项目结构
4. 识别性能预算和质量要求

**检查清单**：
- [ ] 技术栈已理解
- [ ] 编码标准已理解
- [ ] 架构模式已理解

### 步骤3：读取相关设计文档

**目标**：理解设计意图和验收标准

**操作方法**：
1. 读取 design/gdd/ 中的相关设计文档
2. 理解相关功能或系统的设计意图
3. 识别验收标准
4. 理解与其他系统的依赖关系

**检查清单**：
- [ ] 相关设计文档已读取
- [ ] 验收标准已识别
- [ ] 依赖关系已理解

### 步骤4：扫描代码库并评估复杂性

**目标**：识别需要更改的代码和复杂性

**操作方法**：
1. 使用 Grep/Glob 识别需要更改的文件和模块
2. 评估受影响文件的代码行数
3. 评估依赖数量和耦合级别
4. 判断是否涉及核心/引擎代码 vs 叶子/功能代码
5. 判断是否可以遵循现有模式或需要新模式
6. 检查受影响区域的现有测试覆盖

**检查清单**：
- [ ] 受影响文件已识别
- [ ] 代码复杂性已评估
- [ ] 集成点已识别
- [ ] 测试覆盖已检查

### 步骤5：读取历史冲刺数据

**目标**：基于历史数据提高估算准确性

**操作方法**：
1. 查找 production/sprints/ 中的类似已完成任务
2. 计算历史速度（计划 vs 实际）
3. 识别估算偏差模式（持续高估或低估）
4. 记录类似任务的估算和实际工作量

**检查清单**：
- [ ] 类似任务已查找
- [ ] 历史速度已计算
- [ ] 偏差模式已识别

### 步骤6：分析风险因素

**目标**：识别可能影响估算的风险

**操作方法**：
1. 评估新技术或不熟悉的库
2. 评估需求清晰度（清楚/模糊）
3. 评估对未完成工作的依赖
4. 评估跨系统集成复杂性
5. 评估性能敏感性

**检查清单**：
- [ ] 技术风险已评估
- [ ] 需求风险已评估
- [ ] 依赖风险已评估

### 步骤7：生成工作量估算（范围）

**目标**：生成带范围的结构化估算

**操作方法**：
1. 始终给出范围（乐观/预期/悲观），从不给出单一数字
2. 乐观估算：一切顺利，无意外，需求明确
3. 预期估算：正常节奏，小问题，一轮审查反馈
4. 悲观估算：出现重大未知，阻塞一天，需求变更
5. 推荐预算应为预期估算（不是最乐观的）
6. 四舍五入到半天增量

**检查清单**：
- [ ] 范围估算已生成（乐观/预期/悲观）
- [ ] 推荐预算已确定
- [ ] 四舍五入到半天增量

### 步骤8：确定置信度水平

**目标**：评估估算的可信度

**操作方法**：
1. **高置信度**：需求明确，系统熟悉，遵循现有模式，之前完成过类似任务
2. **中置信度**：一些未知，涉及中等复杂系统，之前工作有部分先例
3. **低置信度**：重大未知，新技术，需求不清楚，或跨多个系统的跨领域关注点
4. 如果置信度低，建议先进行时间框定的探索

**检查清单**：
- [ ] 置信度已确定
- [ ] 低置信度时已建议探索

### 步骤9：输出结构化估算报告

**目标**：生成完整的估算报告

**操作方法**：
1. 输出任务描述重述
2. 输出复杂性评估表
3. 输出工作量估算表（乐观/预期/悲观）
4. 输出推荐预算和置信度
5. 输出风险因素表
6. 输出依赖关系表
7. 输出建议分解（子任务）
8. 输出历史对比（如有）
9. 输出备注和假设（明确包含/排除的内容）

**检查清单**：
- [ ] 复杂性评估已输出
- [ ] 工作量估算已输出
- [ ] 置信度已输出
- [ ] 风险因素已输出
- [ ] 依赖关系已输出
- [ ] 建议分解已输出
- [ ] 假设已明确

## 操作注意事项

### 范围估算的重要性

1. **始终给出范围**：从不给出单一数字，单点估算产生虚假精度
2. **推荐预期估算**：推荐预算应为预期估算，不是最乐观的
3. **填充是现实**：填充不是不诚实，而是对未知因素的现实考虑

### 范围边界的明确性

1. **明确包含内容**：明确说明估算包含什么（编码、测试、审查）
2. **明确排除内容**：明确说明估算不包含什么（部署、文档）
3. **范围模糊是错误来源**：范围模糊是估算错误的最常见来源

### 历史数据的正确使用

1. **查找类似任务**：查找真正类似的任务，而非表面相似
2. **识别差异**：记录使任务相似或不同的原因
3. **调整偏差模式**：如果历史显示持续高估或低估，调整当前估算

### 低置信度的处理

1. **建议探索**：如果置信度低，建议先进行时间框定的探索
2. **不要默默填充**：不要默默填充估算，明确指出风险因素
3. **分阶段估算**：对于大任务，考虑分阶段估算（先探索，再实现）

## 协作协议

### 向谁汇报

- **producer**：估算结果和置信度
- **technical-lead**：复杂性评估和技术风险
- **scrum-master**：估算输入到冲刺规划

### 协调对象

- **gameplay-programmer**：验证代码复杂性评估
- **game-designer**：确认设计文档理解
- **qa-lead**：确认测试覆盖需求

## 常见错误

### 错误1：给出单一数字估算

**问题**：输出"3天"而非"2-4天"

**修正**：始终给出范围（乐观/预期/悲观），从不给出单一数字

### 错误2：推荐最乐观估算

**问题**：推荐预算使用乐观估算（"最好情况"）

**修正**：推荐预算应为预期估算（"正常情况"）

### 错误3：忽略历史数据

**问题**：不查看历史类似任务，凭空估算

**修正**：查找历史类似任务，计算历史速度，识别偏差模式

### 错误4：范围边界不明确

**问题**：不说明估算包含/排除的内容

**修正**：明确说明包含什么（编码、测试、审查）和排除什么（部署、文档）

### 错误5：低置信度时不建议探索

**问题**：置信度低但仍给出具体估算，不建议先探索

**修正**：如果置信度低，建议先进行时间框定的探索（如"2天探索，然后重新估算"）
