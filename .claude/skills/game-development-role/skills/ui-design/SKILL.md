# UI Design

设计游戏用户界面，包括界面布局、交互设计、视觉风格和可访问性，确保UI直观易用。

## 适用场景

- 需要为某个玩法/系统设计菜单、HUD、弹窗、信息面板等 UI
- 需要把玩法状态、资源、冷却、任务等信息转成可理解、可操作的界面
- UI 出现可用性问题（误触、找不到入口、信息过载、反馈不清）需要迭代
- 需要与程序/UX/美术/策划协作，明确界面数据、交互规则与验收标准

## 操作步骤

### 步骤1：把 UI 需求改写为“界面清单 + 验收标准”

**目标**：把“要做一套UI”落到可交付、可验收的范围

**操作方法**：
1. 列 UI 入口清单（从哪里进入、退出回到哪里）
2. 列界面清单（菜单/HUD/弹窗/面板）与每个界面的核心任务（用户来这里要做什么）
3. 列每个界面的数据字段（显示哪些数值/状态/列表）
4. 写 3-5 条验收标准（例如：3步内能完成核心操作；关键状态不依赖颜色单一表达）

**检查清单**：
- [ ] 范围明确：有哪些界面/不做哪些界面
- [ ] 字段明确：每个界面显示的数据来自哪里
- [ ] 验收明确：可测试、可复现

### 步骤2：设计信息架构（让用户找得到）

**目标**：建立清晰的层级与导航，避免“入口散落/层级混乱”

**操作方法**：
1. 画导航结构（一级/二级页面、面包屑、返回规则）
2. 定义统一的命名与分组（同类信息放一起，避免同义词）
3. 为高频操作建立固定入口（例如 HUD 快捷按钮、常驻入口）
4. 约定“弹窗策略”（确认/提示/中断/弱提示）

**检查清单**：
- [ ] 从任意界面都能回到安全位置（主菜单/上一级/游戏内）
- [ ] 关键功能入口不超过 2 层
- [ ] 弹窗不会无序叠加

### 步骤3：做线框图并定义交互规则

**目标**：先对齐布局与交互，再进入视觉细化

**操作方法**：
1. 按“主任务优先”摆放信息：主操作区、状态区、次要信息
2. 标注交互：点击/长按/拖拽/悬停/键盘/手柄焦点移动
3. 为列表/背包/商店等组件定义状态：空/加载中/错误/无权限/已售罄
4. 标注输入限制与容错（防误触、二次确认、撤销/回滚）

**检查清单**：
- [ ] 交互路径完整（含失败路径）
- [ ] 列表/表单有空态与错误态
- [ ] 关键操作有防误触策略

### 步骤4：定义反馈与动效（让用户知道发生了什么）

**目标**：为每个操作提供即时且可理解的反馈

**操作方法**：
1. 为交互定义反馈层级：即时反馈（按钮状态）/结果反馈（toast）/系统反馈（加载/错误）
2. 标注关键动效触发点（获得道具、升级、冷却完成、任务完成）
3. 定义音效/震动（如果平台支持）与触发规则
4. 写“错误提示文案规则”（可行动、可解决）

**检查清单**：
- [ ] 每个操作都有“进行中/成功/失败”反馈
- [ ] 动效不会干扰操作（可跳过/可关闭）
- [ ] 错误提示包含下一步行动

### 步骤5：做可访问性与多输入适配

**目标**：保证不同输入方式与可访问性需求下可用

**操作方法**：
1. 为颜色编码补充非颜色信息（图标/文本/形状）
2. 校验对比度与字号层级（关键文本可读）
3. 设计键盘/手柄焦点路径（可达、可见、可预测）
4. 适配分辨率与安全区（不同屏幕比例/刘海/overscan）

**检查清单**：
- [ ] 关键信息不依赖颜色单一表达
- [ ] 焦点路径完整且高亮清晰
- [ ] 不同分辨率下布局不溢出

### 步骤6：输出 UI 规格并对接实现

**目标**：让程序与美术能直接落地，减少返工

**操作方法**：
1. 输出界面清单与状态机（页面/弹窗/组件状态）
2. 输出字段表（字段名、类型、来源、刷新频率、格式化规则）
3. 输出交互规格（点击范围、热区、快捷键/手柄映射）
4. 输出切图/组件规范（9-slice、字体、图标、间距、对齐）

**检查清单**：
- [ ] 字段与接口对齐（命名一致）
- [ ] 状态齐全（加载/空/错误/禁用）
- [ ] 交互与输入映射明确

## 操作注意事项

### 一致性

1. 同一操作在不同界面使用同一布局与文案（例如“确认/取消”的位置与命名）。
2. 同一资源/状态不要出现多个同义词（例如“金币/金钱/货币”统一）。

### 反馈与容错

1. 高风险操作必须二次确认或可撤销。
2. 错误提示要可行动：告诉用户“如何恢复/如何重试/找谁处理”。

### 性能与可读性

1. 列表与滚动容器避免一次性渲染过多元素（需要虚拟列表时提前标注）。
2. HUD 信息控制密度：只显示当前决策需要的信息。

## 输出格式

### UI 交付物

- 界面清单（页面/弹窗/HUD）与入口说明
- 信息架构与导航规则图
- 线框图（含交互标注）
- 视觉稿/组件库（颜色、字体、间距、按钮状态）
- UI 状态机（正常/加载/空/错误/禁用）
- 字段表（字段名、来源、刷新频率、格式化）
- 交互与输入映射表（鼠标/键盘/手柄/触摸）

## 协作协议

### 向谁汇报

- lead-programmer：数据字段可实现性、UI 架构约束
- technical-director：跨系统交互与技术风险

### 协调对象

- ui-programmer：组件实现、动效与交互落地
- gameplay-programmer：数据来源与事件触发
- level-designer：HUD 信息在关卡节奏中的呈现
- qa-tester：可用性回归与多输入测试

## 常见错误

### 错误1：只做视觉稿，没有字段表与状态机

**问题**：实现时字段命名/刷新逻辑不清，返工频繁。

**修正**：按步骤6补齐字段表、状态机与输入映射。

### 错误2：缺少失败路径与空态设计

**问题**：网络异常/无数据时界面崩坏或不可操作。

**修正**：在步骤3为每个列表/表单补齐加载/空/错误/禁用状态。

### 错误3：关键信息只靠颜色表达

**问题**：色弱/不同屏幕下不可读，导致误操作。

**修正**：在步骤5增加图标/文本/形状等冗余表达，并校验对比度。
