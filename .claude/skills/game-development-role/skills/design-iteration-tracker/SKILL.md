# Design Iteration Tracker

跟踪变化内容、变化原因和结果。防止循环设计和构建机构记忆。

## 为什么跟踪迭代？

### 防止循环设计
"让我们尝试让战斗更快。"
"我们在v3尝试过。玩家感到匆忙无法读取棋盘状态。我们在v4因此原因将其放慢。"

### 保留推理
六个月后，为什么这个值设为7？因为它是5时玩家感到匮乏，然后是10时玩家感到泛滥，然后是7测试良好。

### 支持知情决策
提议变更时，知道已尝试的内容。

### 入门新贡献者
新团队成员不仅理解设计是什么，还理解它如何到达那里。

## 核心交付物

### 1. 决策日志

主要记录。每个有意义的设计变更都有条目：

```
决策ID: [唯一标识符——例如 DEC-042]
日期: [何时]
变更: [具体改变了什么]
从: [先前状态/值]
到: [新状态/值]
理由: [为什么进行此变更]
数据源: [什么证据支持此？玩测、直觉、数据？]
影响技能: [哪些设计领域受影响？]
决定者: [谁做出决定？]
状态: [已实现 / 待定 / 已还原 / 已取代]
```

**决策条目的质量规则：**
- 精确陈述变更（"将起始生命从100增加到150"而非"调整生命"）
- 始终记录理由，即使是"设计师直觉"——有效但应明确
- 存在玩测数据时链接（PT-007发现玩家感到太脆弱）
- 记录决定者——为了问责和未来咨询

### 2. 还原警告

最有价值的功能。检测提议变更是否移向先前拒绝的状态：

```
警告：提议变更 [X] 类似于已还原决策 DEC-023。

DEC-023 尝试：[尝试的内容]
DEC-023 结果：[发生什么——为什么被还原]
DEC-023 数据：[什么证据驱动还原]

如果继续：[使此次不同的条件]
如果中止：[使此次相同错误的条件]
```

**"此次不同"检查：**
1. 上下文是否改变？（新系统、不同玩家群、不同平台）
2. 提议实现是否不同？（相同方向但不同执行）
3. 原始还原是否数据驱动或直觉驱动？
4. 我们能在提交前廉价测试吗？

### 3. 影响链跟踪

当变更在设计中级联时：

```
变更：[初始变更——例如 DEC-042：将起始生命增加到150]

直接影响：
  → 平衡：所有伤害值可能需要重新缩放
  → 经济：生命药水价值相对于生命池改变
  → 遭遇：早期遭遇难度转移

间接影响：
  → 进度：生命升级影响较小（较小%增益）
  → 构建多样性：坦克构建与基线区分较小

监控但尚未行动：
  → 玩家体验：此变更是否改变早期游戏情感弧线？
```

### 4. 参数版本历史

对于关键设计值，维护历史：

```
参数：起始玩家生命
────────────────────────────────────────
v1 (初始):     100    [设计师估算]
v2 (PT-003):   80     [玩家感到太安全，减少以增加紧张感]
v3 (PT-005):   120    [80太惩罚，过度校正]
v4 (PT-007):   100    [返回基线，问题是伤害而非生命]
v5 (PT-009):   150    [伤害重新平衡，生命增加以构建多样性]
当前:          150
```

这对避免"调整跑步机"——来回调整相同值而无进展——非常宝贵。

### 5. 迭代摘要报告

设计演变的定期摘要：

```
迭代期间：[日期范围或玩测范围]
总变更：[计数]
主要变更：[列出重要的]
还原：[任何已撤销的变更]
开放问题：[什么仍未解决？]
设计方向：[游戏方向如何转移？]
下一步优先级：[下次迭代应关注什么？]
```

## 跟踪方法论

### 跟踪什么

**始终跟踪：**
- 核心机制变更（规则、值、系统）
- 支柱或愿景变更（最高影响变更）
- 玩测驱动变更（链接玩测数据）
- 还原（为什么撤销某些内容）

**重要时跟踪：**
- 内容变更（新关卡、遭遇、角色）
- 平衡调整（带理由的值调整）
- 美学变更（艺术方向转移、音频变更）

**不跟踪：**
- 拼写修复和格式
- 进行中状态（仅决策时跟踪）
- 未达到共享设计的个别开发者实验

### 如何分类

**变更类型：**
- 添加——设计中的新元素
- 移除——从设计中删除元素
- 修改——现有元素改变
- 还原——先前变更撤销
- 重新平衡——现有值的定量调整
- 重组——组织变更（系统重组，非修改）

**变更驱动：**
- 玩测数据——来自测试的可观察证据
- 玩家反馈——测试者报告的体验
- 设计师洞察——专业判断
- 技术约束——实现强制变更
- 范围压力——时间/预算强制削减
- 市场转移——外部因素改变优先级

## 反模式

### 未记录变更
"哦，我上周改了那个。"无人知道为什么。无人能评估影响。

### 遗忘循环
每几个月尝试相同的失败方法，因为无人记得为什么被拒绝。

### 跟踪税
文档繁重导致设计师停止更新。保持条目简洁。

### 合理化剧场
变更后写理由以证明其合理，而非变更前评估。

### 冻结设计
使用变更日志作为不变更的理由。"我们已经尝试过"不应结束讨论，如果上下文已改变。

## 工作流程

### 记录设计变更

1. 记录当前状态——它现在是什么？
2. 记录提议变更——它将变成什么？
3. 陈述理由——为什么此变更？
4. 检查还原警告——我们之前尝试过吗？
5. 映射影响链——还有什么受影响？
6. 实现变更
7. 更新参数历史（如适用）
8. 标记受影响技能到一致性引擎

### 运行迭代审查

1. 收集自上次审查以来的所有变更
2. 按类型和驱动分类
3. 识别模式（我们是否不断调整相同内容？）
4. 检查循环变更（A→B→A模式）
5. 评估整体设计方向——游戏收敛还是发散？
6. 设置下次迭代优先级

## 输出

此技能产生：
1. 决策日志——每个变更带理由、数据和影响
2. 还原警告——循环设计模式检测
3. 影响链跟踪——变更的级联效应
4. 参数版本历史——关键设计值的演变
5. 迭代摘要报告——定期设计演变概述

这些输出馈入：
- 所有技能——任何设计元素都可以改变
- 平衡技能——平衡是最常迭代的元素
- GDD技能——GDD必须反映当前状态
- 一致性技能——馈送引擎的变更感知
