# Gate Check

验证准备进入下一个开发阶段。生成通过/关注/失败的决策，包含具体的阻塞和必需的工件。

## 适用场景

- 验证项目是否准备好从概念进入预生产
- 验证项目是否准备好从预生产进入生产
- 验证项目是否准备好从生产进入 Alpha
- 验证项目是否准备好从 Alpha 进入 Beta
- 验证项目是否准备好从 Beta 进入发布
- 需要生成阶段转换的结构化检查报告

## 操作步骤

### 步骤1：确定目标阶段门

**目标**：明确要验证的阶段转换

**操作方法**：
1. 解析参数：
   - 带参数：如 `/gate-check production` 验证特定阶段准备度
   - 无参数：自动检测当前阶段，验证下一阶段转换
2. 确定目标门（概念→预生产、预生产→生产、生产→Alpha、Alpha→Beta、Beta→发布）
3. 记录当前日期和检查者

**检查清单**：
- [ ] 目标门已确定
- [ ] 当前阶段已识别
- [ ] 检查日期已记录

### 步骤2：检查必需工件

**目标**：验证所有必需工件存在且有真实内容

**操作方法**：
1. 使用 Glob 和 Read 验证文件存在
2. 验证文件有真实内容（不只是模板标题）
3. 对于代码检查，验证目录结构和文件计数
4. 根据目标门检查具体工件：
   - **概念→预生产**：game-concept.md、游戏支柱、引擎选择、技术偏好
   - **预生产→生产**：systems-index.md、GDD、架构决策、原型、引擎参考、冲刺计划
   - **生产→Alpha**：src/代码、核心机制实现、测试、玩测报告
   - **Alpha→Beta**：功能实现、内容完成、本地化、QA计划、平衡审查
   - **Beta→发布**：发布清单、商店元数据、变更日志、支持计划

**检查清单**：
- [ ] 所有必需文件已检查
- [ ] 文件内容已验证（非空）
- [ ] 代码结构已验证（如适用）

### 步骤3：运行质量检查

**目标**：验证工件的实际质量

**操作方法**：
1. **测试检查**：通过 Bash 运行测试套件（如果配置）
2. **设计审查检查**：读取 GDD 并检查8个必需部分
3. **性能检查**：读取 technical-preferences.md 并与性能分析数据比较
4. **本地化检查**：在 src/ 中搜索硬编码字符串
5. **错误检查**：检查错误跟踪器中的关键/阻塞错误
6. 根据目标门执行特定质量检查

**检查清单**：
- [ ] 测试已运行（如适用）
- [ ] 设计审查已验证（如适用）
- [ ] 性能已检查（如适用）
- [ ] 本地化已检查（如适用）
- [ ] 错误状态已检查

### 步骤4：执行交叉引用检查

**目标**：验证文档与实现的一致性

**操作方法**：
1. 比较 design/gdd/ 文档与 src/ 实现
2. 检查架构文档中引用的每个系统都有对应代码
3. 验证冲刺计划引用真实工作项
4. 检查 GDD 验收标准是否与实际实现匹配

**检查清单**：
- [ ] 设计与实现已交叉验证
- [ ] 架构与代码已交叉验证
- [ ] 计划与实际已交叉验证

### 步骤5：协作评估无法自动验证的项目

**目标**：获取人工验证的项目状态

**操作方法**：
1. 识别无法自动验证的项目（如用户体验、核心循环感受）
2. 询问用户这些项目的状态
3. 从不假设通过，标记为需要手动检查
4. 记录用户反馈

**检查清单**：
- [ ] 无法自动验证的项目已识别
- [ ] 用户已询问
- [ ] 状态已记录

### 步骤6：生成决策和报告

**目标**：输出结构化检查报告和明确决策

**操作方法**：
1. 输出必需工件状态（X/Y 存在）
2. 输出质量检查状态（X/Y 通过）
3. 列出所有阻塞（必须解决才能前进）
4. 列出建议（可选改进）
5. 做出明确决策：
   - **通过**：所有必需工件存在，所有质量检查通过
   - **关注**：存在小差距但可在下一阶段解决
   - **失败**：必须解决关键阻塞才能前进
6. 基于决策提供具体后续步骤

**检查清单**：
- [ ] 工件状态已输出
- [ ] 质量状态已输出
- [ ] 阻塞已列出
- [ ] 决策已明确
- [ ] 后续步骤已建议

## 操作注意事项

### 工件检查的严格性

1. **不只检查存在性**：文件存在不代表有内容，必须验证真实内容
2. **不只检查模板**：确保文件不是空模板，有实际内容填充
3. **代码结构验证**：对于代码检查，验证目录结构和文件组织

### 质量检查的客观性

1. **运行实际测试**：不要假设测试通过，必须运行测试套件
2. **检查实际性能**：与性能分析数据比较，不依赖主观评估
3. **检查实际错误**：查询错误跟踪器，不假设无错误

### 决策的明确性

1. **通过**：所有必需工件存在，所有质量检查通过
2. **关注**：存在小差距但可在下一阶段解决（需说明具体差距）
3. **失败**：必须解决关键阻塞才能前进（列出所有阻塞）

### 阶段门的不可跳过性

1. **顺序性**：必须按顺序通过每个阶段门，不能跳过
2. **阻塞性**：失败的阶段门必须解决阻塞后才能重新检查
3. **记录性**：每次门控检查结果必须记录在项目文档中

## 协作协议

### 向谁汇报

- **producer**：门控检查结果和决策
- **technical-director**：技术工件和质量检查状态
- **qa-lead**：质量检查和测试状态

### 协调对象

- **lead-designer**：设计文档和GDD的准备度
- **technical-lead**：架构决策和代码实现状态
- **qa-tester**：测试执行和错误验证

## 常见错误

### 错误1：只检查文件存在性

**问题**：只检查文件是否存在，不验证文件内容

**修正**：使用 Read 工具验证文件有真实内容，不只是模板标题

### 错误2：假设质量检查通过

**问题**：不运行实际测试，假设测试通过

**修正**：必须通过 Bash 运行测试套件，记录实际结果

### 错误3：决策模糊

**问题**：输出"基本可以"而非明确的"通过/关注/失败"

**修正**：必须使用标准决策术语，明确说明阻塞或差距

### 错误4：忽略无法自动验证的项目

**问题**：跳过需要人工验证的项目（如用户体验）

**修正**：识别这些项目，询问用户，标记为需要手动检查

### 错误5：阻塞不具体

**问题**：只说"需要修复"而不说明具体修复什么

**修正**：每个阻塞必须具体（如"tests/unit/ 中3个失败测试"）
