# Gate Check

验证准备进入下一个开发阶段。生成通过/关注/失败的决策，包含具体的阻塞和必需的工件。

## 阶段门定义

### 门：概念 → 预生产
**必需工件：**
- [ ] design/gdd/game-concept.md 存在且有内容
- [ ] 游戏支柱定义（在概念文档或 design/gdd/game-pillars.md 中）
- [ ] 引擎选择（CLAUDE.md 技术栈不是 [CHOOSE]）
- [ ] 技术偏好配置（.claude/docs/technical-preferences.md 已填充）

**质量检查：**
- [ ] 游戏概念已审查（/design-review 决策不是需要重大修订）
- [ ] 核心循环已描述和理解
- [ ] 目标受众已识别

### 门：预生产 → 生产
**必需工件：**
- [ ] 系统索引存在于 design/gdd/systems-index.md，至少枚举了 MVP 系统
- [ ] design/gdd/ 中至少有1个GDD（除 game-concept.md 和 systems-index.md 外）
- [ ] docs/architecture/ 中至少有1个架构决策记录
- [ ] prototypes/ 中至少有1个原型带 README
- [ ] docs/engine-reference/ 中存在引擎参考文档
- [ ] production/sprints/ 中存在第一个冲刺计划

**质量检查：**
- [ ] GDD 通过设计审查（8个必需部分存在）
- [ ] 原型验证核心循环假设
- [ ] 架构决策覆盖核心系统（渲染、输入、状态管理）
- [ ] 技术偏好设置了命名约定和性能预算

### 门：生产 → Alpha
**必需工件：**
- [ ] src/ 有按子系统组织的活动代码
- [ ] GDD 中的所有核心机制已实现（交叉引用 design/gdd/ 和 src/）
- [ ] 主要游戏玩法路径可端到端游玩
- [ ] tests/ 中存在测试文件
- [ ] 至少有1个玩测报告（或已运行 /playtest-report）

**质量检查：**
- [ ] 测试通过（通过 Bash 运行测试套件）
- [ ] 任何错误跟踪器或已知问题中没有关键/阻塞错误
- [ ] 核心循环按设计游玩（与 GDD 验收标准比较）
- [ ] 性能在预算内（检查 technical-preferences.md 目标）

### 门：Alpha → Beta
**必需工件：**
- [ ] 里程碑计划中的所有功能已实现
- [ ] 内容完成（设计文档中引用的所有关卡、资产、对话存在）
- [ ] 本地化字符串已外部化（src/ 中没有硬编码的面向玩家的文本）
- [ ] 存在 QA 测试计划
- [ ] 平衡数据已审查（运行 /balance-check）

**质量检查：**
- [ ] 所有测试通过
- [ ] 性能目标达成（帧率、内存、加载时间）
- [ ] 没有关键或高严重性错误
- [ ] 覆盖可访问性基础（如适用，重新映射、文本缩放）
- [ ] 保存/加载系统可靠工作

### 门：Beta → 发布
**必需工件：**
- [ ] 发布清单完成（运行 /release-checklist 或 /launch-checklist）
- [ ] 商店元数据准备（如适用）
- [ ] 起草变更日志/补丁说明
- [ ] 存在发布后支持计划

**质量检查：**
- [ ] QA 全面通过 qa-lead 签署
- [ ] 所有目标平台达到性能目标
- [ ] 没有已知的关键、高或中等严重性错误
- [ ] 所有目标语言本地化已验证
- [ ] 满足法律要求（EULA、隐私政策、年龄评级，如适用）
- [ ] 构建干净编译和打包

## 门控检查要素

### 1. 工件检查 (Artifact Checks)
- 使用 Glob 和 Read 验证文件存在且有有意义内容
- 不只检查存在性——验证文件有真实内容（不只是模板标题）
- 对于代码检查，验证目录结构和文件计数

### 2. 质量检查 (Quality Checks)
- 测试检查：通过 Bash 运行测试套件（如果配置了测试运行器）
- 设计审查检查：读取 GDD 并检查8个必需部分
- 性能检查：读取 technical-preferences.md 并与 tests/performance/ 或最近的 /perf-profile 输出中的性能分析数据比较
- 本地化检查：在 src/ 中搜索硬编码字符串

### 3. 交叉引用检查 (Cross-Reference Checks)
- 比较 design/gdd/ 文档与 src/ 实现
- 检查架构文档中引用的每个系统都有对应代码
- 验证冲刺计划引用真实工作项

## 门控检查原则

### 1. 全面性
- 检查所有必需工件
- 检查所有质量标准
- 检查所有阻塞

### 2. 严格性
- 不只检查存在性
- 验证真实内容
- 验证实际质量

### 3. 协作性
- 询问无法自动验证的项目
- 从不假设通过
- 用户做最终决定

### 4. 可操作性
- 提供具体阻塞
- 提供具体建议
- 提供后续步骤

## 门控检查流程

### 1. 解析参数
- 带参数：/gate-check production — 验证特定阶段准备度
- 无参数：自动检测当前阶段，然后验证下一阶段转换

### 2. 运行门控检查
- 对于目标门中的每个项目
- 工件检查
- 质量检查
- 交叉引用检查

### 3. 协作评估
- 询问用户无法自动验证的项目
- 从不假设通过
- 标记为需要手动检查

### 4. 输出决策
- 输出必需工件状态
- 输出质量检查状态
- 输出阻塞
- 输出建议
- 输出决策

### 5. 后续行动
- 基于决策建议具体后续步骤
- 提供可操作建议
- 提供相关技能

## 输出格式

### 门控检查报告
```markdown
## 门控检查：[当前阶段] → [目标阶段]

**日期**：[日期]
**检查者**：gate-check 技能

### 必需工件：[X/Y 存在]
- [x] design/gdd/game-concept.md — 存在，2.4KB
- [ ] docs/architecture/ — 缺失（未找到ADR）
- [x] production/sprints/ — 存在，1个冲刺计划

### 质量检查：[X/Y 通过]
- [x] GDD 有 8/8 必需部分
- [ ] 测试 — 失败（tests/unit/ 中3个失败）
- [?] 核心循环已玩测 — 需要手动检查

### 阻塞
1. **没有架构决策记录** — 运行 /architecture-decision 创建一个
   涵盖核心系统架构，然后进入生产。
2. **3个测试失败** — 在前进之前修复 tests/unit/ 中的失败测试。

### 建议
- [解决阻塞的优先行动]
- [不是阻塞的可选改进]

### 决策：[通过 / 关注 / 失败]
- **通过**：所有必需工件存在，所有质量检查通过
- **关注**：存在小差距但可在下一阶段解决
- **失败**：必须解决关键阻塞才能前进
```
