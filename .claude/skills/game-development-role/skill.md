# Game Development Role Skill

帮助用户扮演专业游戏开发角色，完成从游戏设计到发布的全流程任务。

## 适用场景

- 用户希望你以“游戏开发团队/游戏开发角色”的方式处理任务，但不确定该由哪个子技能/子角色来执行
- 需要将一个较大任务拆解为多个可执行的游戏开发子任务，并按顺序调用子技能
- 需要在任务执行前明确目标平台/引擎/限制条件，从而选择正确的实现路径

## 使用方式

本节描述如何使用本 skill 作为“任务路由器”，将用户请求转换为可执行的子技能调用计划，并逐步交付。

### 触发方式

当用户使用以下任一触发词时，本skill将被激活：
- "扮演游戏开发"
- "游戏开发角色"
- "游戏开发任务"
- "游戏设计"
- "游戏编程"
- "游戏美术"
- "游戏QA"
- "Godot开发"

### 工作流程

1. **收集输入**：让用户补齐必要信息，确保能落到可执行层
2. **确定执行路径**：将任务路由到 1-N 个子技能（设计/编程/美术/QA/制作）
3. **逐步执行**：每次只推进一个子任务，输出阶段性可验收结果
4. **汇总交付**：将各子技能产出整理为最终交付物清单与下一步建议

**检查清单**：
- [ ] 已明确目标：做什么、完成标准是什么
- [ ] 已明确约束：引擎/平台/性能预算/截止时间
- [ ] 已明确输入：已有资源、仓库/项目路径、参考链接
- [ ] 已选择子技能：并说明选择理由

## 操作步骤

### 步骤1：将用户请求改写为可路由的任务描述

**目标**：把“我想做个游戏/我需要实现XX”改写为可执行、可拆解、可验收的任务

**操作方法**：
1. 用 1-2 句复述用户目标（避免引入新需求）
2. 明确交付物类型（文档/实现/资产/测试报告/发布包等）
3. 列出硬约束：引擎、平台、语言、性能预算、截止时间
4. 列出缺失信息，并提出不超过 5 个澄清问题

**检查清单**：
- [ ] 目标可验收（有完成标准）
- [ ] 约束已明确
- [ ] 依赖/输入已明确（仓库、素材、参考）

### 步骤2：选择 1-N 个子技能并制定执行顺序

**目标**：把任务路由到合适的子技能，并确定先后顺序

**操作方法**：
1. 优先选择“最接近交付物”的子技能
2. 若任务跨域（例如设计 + 实现 + QA），按“先定义再实现、先最小可用再优化”的顺序编排
3. 对每个子技能，写清楚它的输入、输出与验收点

**检查清单**：
- [ ] 子技能选择与目标匹配
- [ ] 顺序能降低返工（先对齐需求/接口/数据）
- [ ] 每一步都有阶段性可验收产出

### 步骤3：执行单步并产出可复用结果

**目标**：推进一个子任务并输出可复用的中间产物

**操作方法**：
1. 明确本步要解决的唯一问题
2. 产出结构化结果（清单/表格/接口约定/代码变更点）
3. 给出下一步调用的子技能与需要的输入

**检查清单**：
- [ ] 本步范围清晰
- [ ] 输出可被下一步直接消费

## 操作注意事项

### 路由与拆解

1. 只把任务拆到"下一步可执行"的粒度，避免一次性拆到过细导致上下文漂移。
2. 若用户没有明确引擎/平台，优先在步骤1补齐信息；如果用户允许探索，用最小代价原型（如 Web/Phaser）先验证。
3. 对每个子技能的调用，都要明确输入、输出、验收点，避免"做了很多但无法验收"。

### 输出一致性

1. 阶段性输出优先使用清单/表格/约定（接口、数据结构、目录结构），确保可复用。
2. 同一轮执行中尽量固定输出格式（例如每步都包含：目标、操作方法、检查清单）。

### 常量引用

1. 在游戏设计相关技能中，可引用 `constants/` 目录下的游戏常量文档
2. 可用常量包括：
   - `GAME_STYLE_CONSTANTS.md` - 游戏风格（国风、欧美、二次元等）
   - `GAME_TYPE_CONSTANTS.md` - 游戏类型（RPG、卡牌、MOBA等）
   - `GAME_THEME_CONSTANTS.md` - 游戏主题（武侠、科幻、赛博朋克等）
   - `UI_FUNCTION_CONSTANTS.md` - UI功能（登录、战斗、商城等）
   - `MATERIAL_PATTERN_CONSTANTS.md` - 材质图案（木头、金属、云纹等）
   - `PLATFORM_CONSTANTS.md` - 平台（手游、PC、主机等）
   - `LAYOUT_CONSTANTS.md` - 版式（横屏、竖屏）
3. 引用格式：`{{CONSTANT_NAME.中文名称}}`，例如 `{{GAME_TYPE_CONSTANTS.角色扮演}}`
4. 使用常量可确保设计文档的一致性和标准化

## 输出格式

### 路由计划（阶段性）

- 用户目标复述（1-2句）
- 约束与假设（引擎/平台/性能预算/截止时间）
- 澄清问题（≤5个）
- 子技能执行计划（按顺序）：每个子技能的输入、输出、验收点

### 交付物清单（最终）

- 已完成的产出列表（文档/代码变更点/资产/测试报告/发布包）
- 未完成项与风险列表
- 下一步建议（按优先级）

## 协作协议

### 与用户的协作

1. 每次推进一个子任务；如果需要跨域协作（设计/编程/美术/QA），先输出“路由计划”再开始实现。
2. 对不确定信息，先提澄清问题；若用户明确接受假设，再继续执行并在输出中标注假设。
3. 若用户提供仓库/项目路径/引擎版本，优先对齐这些约束，避免产生不可落地的建议。

## 常见错误

### 错误1：没有明确引擎/平台就开始实现

**问题**：会导致实现路线、资源格式、性能预算等关键决策反复变更。

**修正**：在步骤1先固定引擎与平台；若不确定，用最小代价方案（如 Web/Phaser 原型）先验证。

### 错误2：一次性调用过多子技能导致产出不一致

**问题**：不同子技能输出格式不统一，汇总时难以验收。

**修正**：按步骤2编排顺序；每步结束先做一次“阶段性交付物清单 + 验收点”。
