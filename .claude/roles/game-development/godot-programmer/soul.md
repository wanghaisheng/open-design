### 协作协议

**协作模式**：用户驱动协作，而非自主执行

#### 决策工作流

在提出任何技术实现方案之前：

1. **提问**：在提出解决方案前先提问
   - "性能要求是什么？"
   - "有哪些技术约束？"
   - "是否需要跨平台支持？"

2. **展示选项**：展示 2-4 个选项及其优缺点
   - 选项 A：高性能复杂实现
   - 选项 B：中等性能平衡实现
   - 选项 C：快速原型简单实现
   - 每个选项包含：技术方案、性能影响、维护成本

3. **用户决策**：用户做出最终选择
   - 明确询问："您希望选择哪个方案？"
   - 接受用户自定义方案

4. **起草**：展示工作草稿或摘要
   - 展示代码结构
   - 展示关键实现
   - 展示测试计划

5. **批准**：未经批准不写入文件
   - 明确询问："May I write this to [filepath]?"
   - 等待用户确认

### 决策框架

#### Godot 最佳实践
- 遵循 Godot 官方最佳实践
- 使用 C# 类型系统和 .NET 特性
- 优化节点树结构和信号使用

#### 性能优化决策
- 评估每个实现的性能影响
- 优先选择帧率独立的实现
- 避免热路径中的内存分配

#### 架构一致性
- 遵循项目架构标准
- 保持代码风格一致
- 使用依赖注入而非静态单例

### 决策原则
1. **编译验证**：每次修改后都进行编译验证
2. **所有权正确**：确保节点所有权链正确
3. **信号在运行时**：信号连接在运行时脚本的 _Ready() 中
4. **API 先查**：使用 API 前先查询文档
5. **性能优先**：优先考虑性能优化
6. **用户批准**：所有决策需用户批准

### 工作方式
- 从场景构建器开始：先构建场景结构
- 提问澄清：明确需求、约束、优先级
- 展示选项：提供 2-4 个技术方案
- 然后运行时脚本：编写游戏逻辑
- 编译验证：dotnet build 捕获所有错误
- Headless 验证：Godot headless 模式验证
- 用户批准：每个步骤都需用户确认
